## Request 1: `v0.17:a11y-prefers-contrast`

Priority: Medium

Goal: add a small `prefers-contrast: more` accessibility CSS pass for
the main editing surfaces without changing layout or feature behavior.

Evidence gap:

- `prefers-reduced-motion` exists, but high-contrast media behavior is
  not implemented.
- App Store quality closeout marks this as low risk but worth a
  manual accessibility pass before submission.

Good outcome:

- Main text/editor chrome, tab state, dialogs, focus rings, status/error
  rows, and workspace affordances remain legible when macOS Increase
  Contrast maps to `prefers-contrast: more`.
- The change is CSS-only unless a focused accessibility test is clearly
  justified.
- Existing theme intent remains recognizable.

Out of scope:

- redesigning themes
- changing color token architecture broadly
- adding new settings
- claiming VoiceOver smoke

Suggested verification:

- `npm run build:vite`
- focused visual/manual note with Increase Contrast enabled if the
  environment supports it
- `git diff --check`

## Request 2: `v0.17:diagnostics-ui`

Priority: Medium

Goal: expose the existing privacy-preserving diagnostics serialization
helper through a small copy/view UI, without telemetry or upload.

Evidence gap:

- The helper exists and has redaction tests, but there is no user-facing
  way to copy diagnostics for support.

Good outcome:

- A reachable support / diagnostics surface displays or copies only the
  already-redacted diagnostic envelope.
- It includes app version, distribution lane, feature flags, and
  sanitized error categories where already available.
- It does not include document contents, provider transcripts, full
  workspace listings, secret-looking values, or unnecessary paths.

Out of scope:

- telemetry
- automatic upload
- crash reporting
- collecting document text
- broad settings redesign

Suggested verification:

- focused unit test for the UI or copy path where practical
- `npm run build:vite`
- `git diff --check`

Close with the same hand-off shape as Request 1.

## Request 3: `v0.17:workspace-tree-role-audit`

Priority: Medium-low

Goal: decide whether the WorkspaceTree should adopt an ARIA `tree`
pattern now, or stay with the current button-based keyboard-operable
structure and document the decision.

Evidence gap:

- The current tree is keyboard-operable, but the closeout notes that a
  formal `role="tree"` pattern is not implemented.
- A partial role change can reduce accessibility quality if focus,
  `aria-expanded`, `aria-level`, and keyboard behavior are incomplete.

Good outcome:

- Either implement the complete minimal ARIA tree pattern with focused
  tests, or add a short documented decision explaining why the current
  button-based model remains preferable for v0.17.
- If implemented, keep file operations, context menus, drag/drop, lazy
  loading, and compare selection behavior intact.

Out of scope:

- replacing the workspace browser with a full file manager
- adding workspace indexing
- adding Git status
- redesigning file operations

Suggested verification:

- focused `WorkspaceTree` tests for keyboard and ARIA behavior if code
  changes
- `npm run build:vite`
- `git diff --check`

Close with the same hand-off shape as Request 1.

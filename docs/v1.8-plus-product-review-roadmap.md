# v1.8+ Product Review And v2 Bridge Roadmap

Status: Accepted planning direction
Scope: v1.8 以降の品質・UX・単一文書構造化、OKF互換preview、v2 Book Scope前の配布版総合テスト
Authority: High for post-v1.7 product sequencing; individual implementation contracts require focused design docs
Last reviewed: 2026-07-16 (v1.12 bridge complete direction; post-v1.12 → refinement roadmap)

## Decision

Hazakura Editor は、v1.7 の直後に v2.0 へ進まない。

`v1.8`、`v1.9`、`v1.10` のような二桁 minor version を通常の継続版として
許容し、v1 系で次の順序を踏む。

1. 現在の Safe Editor / Reference Compare を日常利用と配布版で信頼できる状態へする。
2. 多い機能名を「書く・読む・確かめる」の利用目的へ整理する。
3. **一つの Markdown source の中で**、章・節・見出し・目次候補を構造として
   確認し、明示的に操作できる文書構造化機能を追加する。
4. OKF v0.1 Draftを、明示的・bounded・read-onlyな互換previewとして確認する。
5. 構造化機能とOKF previewを含む配布版で、長文・復旧・参照・書き出し・
   アクセシビリティを横断する大きめの総合テストを改めて行う。
6. その証拠を得た後に、複数 Markdown ファイルを一冊として扱う v2 Book Scope
   の実装契約を確定する。

この順序は version number の短さより、製品の成熟と安全な段階移行を優先する。
必要なら `v1.11`、`v1.12` 以降も使用してよい。`v2.0` は単なる次番号ではなく、
単一文書から複数文書 Book Scope へ製品境界が広がる時だけ使用する。

## Product Review Summary

### Preserve

- Markdown / text source を保存の正本とする。
- 編集可能な主バッファは一つに保つ。
- Preview、L Mode、e-book、Reference、Diff、Recovery、Local Assist を
  source-preserving な解釈・確認レイヤーとして扱う。
- OCR / AI / recovery candidate は自動保存・自動適用せず、利用者が確認する。
- Safe Editor と Agent Workbench の trust boundary を分離する。
- workspace 操作、PDF raster、検索、Diff、草稿保存へ個別の上限と失敗表示を持つ。

### Improve Before Breadth

- full Rust suite の host-sensitive failure を決定的にする。
- pathless recovery、長い Reference、keyboard / VoiceOver、sandbox picker を
  配布版で証明する。
- Preview / Reference / e-book / Outline / Diff / L Mode / Assist の違いを、
  内部機能名ではなく利用目的から理解できるようにする。
- `useAppShellController` に集中した document session、presentation、review
  transaction の配線を、挙動を変えない範囲で観測・分割可能にする。
- main bundle の大きさだけでなく、入力遅延、Preview sanitize、pagination、Diff、
  local draft persistence、PDF raster memory を計測する。
- macOS deployment target と Local Assist availability を同じ判断にしない。

## Recommended Version Sequence

Version numbers are planning labels, not release promises. A lane may be split into
additional `v1.xx` versions when verification or reviewability benefits from a smaller slice.

### v1.8 — Daily Trust Completion

Goal: v1.7 までに実装された体験を、source green だけでなく配布版で信頼できる状態へする。

Primary outcomes:

- full Rust suite の bookmark / Trash / PATH-sensitive tests を隔離し、全体実行を決定的にする。
- pathless draft forced-termination recovery を packaged app で確認する。
- Reference Compare の picker / sandbox / CSP / source deletion / replacement / narrow windowを確認する。
- Quick Open、Command Palette、Global Search、rename、modal、pane controls の
  keyboard / VoiceOver semantics を整える。
- 5,000-line wrapped reference と budget overrun の失敗表示を確認する。

Non-goal: 新しい大きな product surface。

### v1.9 — Writing Loop Clarity (source complete; rolls forward with v1.11)

Goal: 「機能を知っている人だけが使える」状態から、書く・読む・確かめる流れで
理解できる状態へ進める。

Primary outcomes:

- top chrome と first-use hints を `書く / 読む / 確かめる` の目的で整理する。
- Start Panel を初回用と再訪用に分け、前回 workspace の再開と復旧候補を明確にする。
- Preview / e-book / Reference / Diff の hidden / retained / closed state を説明する。
- L Mode 往復、tab切替、Save As後も利用者の作業文脈を保つ。
- command palette の日本語 / かな discoverability を揃える。

Non-goal: toolbar の増築、汎用 split editor、設定項目の大量追加。

### v1.10 — Single-document Structure Foundation (TestFlight candidate prepared)

Goal: v2 Book Scope の前に、**単一 Markdown 文書内の構造**を利用者が確認し、
必要な場合だけ明示的に整えられるようにする。

Implemented source shape:

- ATX headings、frontmatter、明示的 page-break marker、EPUB navigation候補を、
  既存 source から一つの shared structure model として解釈する。
- chapter / section hierarchy を source位置付きで表示し、クリックでEditorへ移動する。
- skipped heading level、空見出し、重複し得るnavigation label、極端に長い章などを
  **助言**として示す。エラーや自動修正とは扱わない。
- e-book TOC、Outline、EPUB navigation が同じ構造解釈を使う。
- 構造表示から行う編集は、単一buffer内の明示的transactionとし、Undo可能・dirty・
  保存前review可能にする。初期対象は section move / heading level change のうち、
  source-preserving proof が得られる最小操作に限定する。

Boundary:

- 一度に編集する Markdown は一つ。
- 第二の保存モデル、独自database、hidden manifestを作らない。
- workspace全体をbackground scanしない。
- 自動章分割、自動並べ替え、自動heading修正、自動保存をしない。
- 複数ファイルの章順管理と Book manifest は v2 に残す。

### v1.11 — OKF Draft Compatibility Preview + Distribution Confidence

Goal: OKF v0.1 Draftを読み取り中心で安全に確認し、文書構造化を含むv1系の
完成形を実際の配布版で横断的に検証する。

このlaneは、限定的なcompatibility previewとrelease-confidenceを組み合わせる。
ユーザーが明示したworkspace root / subfolderを一度だけbounded scanし、OKF
concept / reserved file / linkの互換性を読み取り専用で示す。Book Scope、章順、
manifest authoring、multi-file edit、auto-repair、background indexingは行わない。
Implementation contract: `docs/v1.11-okf-draft-preview-design.md`.
Shared pin / upgrade process: `docs/okf-spec-pin.md`.

### v1.12 — OKF Starter Scaffold

Goal: 明示操作で OKF 互換の Markdown 群ひな形を置く（最小 / 本っぽい章立て）。
read の v1.11 と **同じ pin** を共有し、upstream OKF が動いたら点検モデルと
テンプレ資産をセットで更新する前提とする。

Implementation contract: `docs/v1.12-okf-scaffold-design.md`.
Pin authority: `docs/okf-spec-pin.md`.

- Entry: Command Palette, folder context menu, workspace **新規 (+)**, OS File menu.
- Write boundary: new unique folder, no overwrite, main window, workspace only.
- Non-goals: Book Scope, auto-repair, multi-file export, formal OKF certification.

#### Golden Structured Manuscript

一つのpublic-safe fixtureに次を含める。

- 長い日本語本文、H1〜H4、frontmatter、page-break markers
- 画像、table、code block、blockquote、task list、長い折返し行
- LF / CRLF、final newline、legacy Japanese encoding の代表例
- Reference Compare用PDF / image / Markdown
- structure warningの正常例と助言例
- OKF valid / invalid concept、`index.md` / `log.md`、unknown field/type、broken link

#### End-to-end Workflow

1. workspace / manuscriptを開く。
2. structure overviewから章へ移動する。
3. 最小の構造編集を行い、Undo / Redo / dirty stateを確認する。
4. Preview / L Mode / e-book / Outline / Reference / Diffを往復する。
5. Save As、external change、dirty close、forced termination recoveryを確認する。
6. Import AssistとReference follow / 要確認を確認する。
7. Local Assist proposalをapply / discardし、sourceへの自動保存がないことを確認する。
8. HTML / PDF / EPUBを書き出し、structure / navigation / imagesを外部readerで確認する。
9. 明示的にOKF reviewを実行し、compatibility summary、助言、concept open、
   cancel、rerun、containment rejectionを確認する。

#### Distribution Layers

- source tests: Vitest、Rust、typecheck、format、audit
- local packaged app: launch、native menus/dialogs、filesystem、helpers
- signed App Store / TestFlight shape: sandbox、entitlements、security-scoped access、helper signing
- manual accessibility: keyboard、VoiceOver、contrast、enlarged text、Reduce Motion、IME
- scale/failure: long document、budget overrun、cancel/retry、memory/latency observations

Passing this gate does not tag, upload, submit, or publish automatically. The
next public App Store release is intentionally deferred until the OKF preview
and this matrix pass; release mutation remains a separate explicit user decision.

## v2 Entry Gate

v2 Book Scope design may move from Proposal toward implementation contract only when:

- v1.10 single-document structure model is stable and shared by Outline / e-book / export;
- the structure editing transaction preserves Undo、dirty、Save As、recovery、IME;
- the v1.11 OKF compatibility preview is bounded, explicit, and does not create
  a persistent index or multi-file edit model;
- the v1.11 distribution-confidence matrix has recorded results and known residual risks;
- the team can explain why a multi-file book adds value that cannot be delivered safely inside one file;
- OKF v0.1 Draft compatibility is recorded without treating OKF as a book-order specification;
- Book Scope UI has one selected direction, not a list of unresolved candidates.

## v2 Remains

v2 is still the place for:

- an explicit, user-selected set of Markdown files treated as one book;
- chapter ordering across files;
- a visible, reversible Book Scope / manifest interpretation;
- whole-book reading and export over multiple source files;
- later vertical-writing reading / export layers after the structural foundation is stable.

v2 is not required merely because the version after `v1.9` would otherwise be `v1.10`.

## Explicit Non-goals Across v1.xx

- Git、LSP、terminal、arbitrary command execution、plugins
- project-wide background indexing or automatic structure detection
- two editable panes or Preview DOM editing
- cloud OCR、generic AI chat、background rewriting
- auto-apply、auto-save、auto-commit、in-app release automation
- a proprietary document database replacing Markdown files
- advanced DTP、PDF editing、full EPUB production suite

## After v1.12

The sequential bridge above ends at OKF starter scaffold + distribution evidence.
Post-v1.12 work is **not** a mandatory v1.13→v1.14→v1.15 package train. Use the
try-and-error refinement roadmap instead:

- `docs/v1.13-plus-refinement-roadmap.md` — themes, experiment loop, hard rails, v2 entry reminder
- `docs/v1.13-interaction-clarity-plan.md` — Theme A (Signal & Silence) candidate pool
- `docs/v1.xx-image-media-boundary-plan.md` — Theme G image/media consent, Preferences, export materialize

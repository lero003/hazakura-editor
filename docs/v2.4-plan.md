Status: Planning (not yet the active implementation queue)
Scope: Next store / source line after published `2.3.0`
Authority: Medium
Last reviewed: 2026-07-24

## Purpose

`2.3.0` は Mac App Store 公開済み、source tag `v2.3.0` も切った。  
次の製品線 **v2.4** は「本としての深み」を一段だけ進め、Safe Editor の境界は広げない。

この文書は **プラン正本** である。実装キューに昇格するまで
`docs/current-work.md` の Active Queue は post-v2.3（hotfix 優先）のまま。

v2.0–v2.3 で閉じた本の基盤:

- Book Scope Alpha（明示選択・app-private 順・単一編集 buffer）

**source tag と main のズレ:** compact Book sidebar は `v2.3.0` タグ後の main に
だけある。tag は動かさない（方針 1）。**v2.4 の store / source 線に含めて揃える。**

## v2.4 Product Intent

一行:

> 選んだ章群を、章単位で確かめ・目次の見え方を整え・静かに書き出す。  
> source は個々の Markdown。indexing / auto-apply / IDE 化はしない。

v2.3 が「持ち運び・続きから読む」なら、v2.4 は **章レビューと目次の深み** を優先する。

## Operating Rules

1. **1 run = 1 verifiable slice.** 失敗テストまたは acceptance を先に固定できるものから。
2. **Published `2.3.0` is hotfix-only** until the v2.4 store line is intentionally opened.
3. **Version bump to `2.4.0`** only when the lane opens (npm / Tauri / Cargo 同時)。今の tree は `2.3.0` のまま polish を積んでよい。
4. **source tag / store close** は別 human gate。tag は immutable。binary は付けない（source 線）。
5. Hard rails: `docs/security-boundary.md` + Agents 表の Surfaces 分類。

## Proposed Lane Opening Sequence

実装を始めるときの目安（自動実行しない）:

```text
[done]  v2.3 MAS + source tag
[main]  C-0 compact Book toolbar (post-tag; fold into 2.4)
[open]  bump package surfaces to 2.4.0 when human opens the lane
[slice] B-1 Chapter Diff  OR  B-2 TOC design contract (pick one first)
[slice] remaining B-* / S-* one at a time
[gate]  local gates → signed pkg → human TestFlight → upload → tag v2.4.0
```

**First slice recommendation when opening v2.4:**

1. バージョン面を `2.4.0` に揃え、C-0 を「v2.4 同梱」と docs に固定する。  
2. 次の機能スライスは **B-1（章 Diff）** を第一候補。  
   - 既存 Diff / dirty / 単一 buffer に乗りやすい  
   - X-4 は v2.x practicalization の次段として明示されている  
3. B-2 は「編集可能 TOC」をいきなり作らず、まず **表示契約と Help/UI の誤解解消** を小さく。編集 UI は契約確定後。

代替: 章 Diff より摩擦が低いなら **B-3 + B-4（説明コピー）** を先に 1 スライスでもよい。  
どちらも「境界を広げない」が条件。

## Open Decisions (human)

実装開始前に決めたいこと（未決のままスライスを始めない）:

1. **第一スライスは B-1（章 Diff）か、B-2（TOC 契約）か、コピー系 B-3/B-4 か。**  
   推奨: B-1。
2. **章 Diff の比較相手** — disk 版のみ / 2 ファイル指定 / 直近 recovery など。  
   既存 Diff 入口を再利用し、新しい Git 的 UI を増やさない。
3. **TOC「編集」の深さ** — v2.4 では表示説明までに留めるか、app-private 表示順の編集まで入れるか。  
   推奨: まず表示契約; 編集 UI は B-2b として分割。

# v1.7 Reference Compare Design

Status: Active product lane / R0–R4 source landed (v1.7 feature arc complete in source)
Scope: Read-only PDF, image, or text reference beside one editable Markdown document
Authority: High for the v1.7 product and interaction boundary
Last reviewed: 2026-07-10 (R4 advisory page review + reliability landed)

## Decision

After **v1.6 (`1.6.0`) passed App Review** and closed Import Assist Phase 1,
v1.7 deepens that intake into a broader **Reference Compare** workflow.

The product concept is:

> PDF、画像、Markdownを「参照」として開き、編集中のMarkdownと横に並べて見比べる。

Import Assist remains an important entry point, but Reference Compare is not
an OCR-only review screen. A user may also open an ordinary Markdown or text
file as a read-only reference and manually compare it with the active editable
Markdown document.

This is intentionally a pair of different roles:

```text
Reference Tab (read-only)  |  Editor Tab (editable)
PDF / image / text         |  canonical Markdown buffer
```

It is **not** a generic split editor and does not introduce two simultaneously
editable document models.

## User Value

v1.6 can turn a PDF or image into an unsaved Markdown draft. The remaining
friction is checking the result: the source document and draft are not visible
together, page context is carried only by source comments, and OCR confidence
is not yet used by the review experience.

Reference Compare closes that gap while also serving non-OCR writing work:

- compare an imported Markdown draft with its source PDF;
- transcribe or revise prose while keeping a scan visible;
- compare two Markdown files by eye without calculating a Diff first;
- keep a style guide, outline, or earlier draft visible while editing;
- move from visual inspection to the existing text Diff only when both sides
  are comparable text.

The v1.7 release story should stay simple:

> 参照資料を横に開き、Markdownを見比べながら直せる。

## Product Boundary

Reference Compare is a **Safe Editor core** surface.

- Markdown/text source remains canonical.
- Only the Editor Tab is editable, dirty, recoverable, and saveable.
- The Reference Tab is read-only and never auto-applies content to the editor.
- Import Assist output remains an unsaved Markdown draft until the user saves.
- References come only from a user-selected file or the selected workspace.
- No Git, LSP, terminal, arbitrary command execution, plugin, project-wide
  indexing, cloud OCR, external viewer automation, or background file scan is
  introduced.
- Reference Compare does not revive the retired standalone Review Desk.
- PDF annotations, form editing, embedded attachments, and JavaScript are not
  executed or exposed.

## Core Interaction Model

### Paired-tab presentation

The existing document Tab Bar remains the primary list of editable/open text
documents. Reference Compare adds a paired presentation inside the document
workspace:

- the reference side has a compact pane-tab header labelled `参照`;
- the editor side keeps the active Editor Tab identity, dirty marker, and
  normal editing behavior;
- a resizer separates the two panes;
- both filenames remain visible so the user always knows what is being
  compared;
- closing the reference returns to the ordinary single-editor layout without
  closing or changing the Editor Tab.

The pane headers should look and behave like two visible tabs, but they are not
two independent global tab groups. This avoids introducing arbitrary split
layouts, tab dragging, or multi-editor state.

### Side order

The default order is:

```text
left: Reference Tab (read-only)
right: Editor Tab (editable Markdown)
```

This matches source-to-draft reading order and the existing Diff convention of
reference/original on the left and target/change on the right. A left/right
swap setting is out of scope for the MVP.

### Entry points

Reference Compare may be entered in three explicit ways:

1. **Import Assist completion**
   - Keep the selected PDF/image as the Reference Tab.
   - Open the generated unsaved Markdown draft as the Editor Tab.
   - Enter Reference Compare automatically after a successful import.
2. **Workspace context menu**
   - `参照として横に開く` on a supported PDF, image, Markdown, or text file.
   - Keep the current editable Markdown as the Editor Tab.
3. **Tab context menu**
   - `参照として横に開く` on an already-open text tab.
   - Use the selected tab as the read-only reference and keep the active
     Markdown as the Editor Tab.
4. **File menu / Command Palette**
   - `参照ファイルを横に開く…` through an explicit native picker.
   - If no editable document is active, open the reference in a read-only
     single-pane state and invite the user to open or create a Markdown file.

Opening a text file as a normal editable tab remains the default action.
`参照として横に開く` is always explicit.

### Reference lifetime

- At most one Reference Tab is visible at a time.
- The reference stays pinned while the user switches among editable tabs.
- Import page-follow behavior is active only for the imported draft session
  that was created from that reference; other tabs show the reference without
  linked-page claims.
- Closing or replacing the Reference Tab does not close, save, or mutate any
  Editor Tab.
- Reference state is in-memory only in the v1.7 MVP. App restart does not
  silently reopen external PDFs or recreate security-scoped access.
- If the referenced file disappears or becomes unreadable, show a bounded
  read-only error state and allow the reference to be closed or replaced.

## Reference Types

### PDF

PDF is displayed as a read-only paged reference, not as an editable or
browser-executed document.

Required controls:

- previous / next page;
- current page and total page count;
- fit page / fit width;
- bounded zoom;
- keyboard-accessible page navigation;
- filename and `読み取り専用` label.

The frontend must not load an ambient `file://` PDF or depend on a browser PDF
plugin. The preferred technical direction is a fixed, app-bundled PDFKit helper
contract that returns document metadata and a bounded raster for the requested
page. Only the current page and a small adjacent-page cache should be held.

The first implementation slice must spike this contract before building the
complete PDF UI. Exact raster dimensions and cache budgets are selected from
the spike, then pinned in tests.

### Image

Reuse the existing safe image loading and `ImagePreviewPane` behavior where
possible.

- fit the image within the reference pane;
- allow bounded zoom and scrolling;
- keep the existing format, signature, size, and workspace-containment rules;
- never import or copy the image merely because it is opened as a reference.

### Markdown / text

Show source text in a selectable, copyable, read-only code surface with line
numbers and the app's existing encoding awareness.

- It is a disk snapshot, clearly labelled `読み取り専用`.
- It does not gain dirty state, Undo history, backup, or save actions.
- If the same file is already the active editable tab, route the user to the
  existing buffer-versus-disk review instead of creating a misleading
  self-reference pair.
- When both sides are text, expose `差分を見る` as a secondary action using the
  existing bounded Diff pipeline. Reference Compare itself remains a visual
  side-by-side view, not a continuously recalculated Diff.

## Import Assist Integration

### Source linking

The frontend already knows the selected source path when import begins. On
success it creates:

- one unsaved Import Assist Editor Tab;
- one read-only Reference Tab for the selected source;
- an in-memory link between the reference and the imported editor `sessionId`.

The link must use `sessionId`, not filename or mutable path identity, so Save As
or tab rename does not attach page-follow behavior to the wrong document.

### Page markers

The existing source-visible markers remain the durable mapping:

```markdown
<!-- hazakura:import-page index=0 -->
```

For the linked imported draft:

- moving the editor cursor into a marked page section updates the PDF reference
  to that page when `追従` is enabled;
- manually changing the PDF page temporarily pauses automatic follow;
- `追従を再開` returns to the page containing the editor cursor;
- if markers are deleted, the editor remains fully editable and Reference
  Compare continues without page-follow claims.

No hidden database or alternate saved document model is introduced.

### Confidence review

Low-confidence review is a later v1.7 slice, after the paired surface and PDF
reader are stable.

The helper contract may be extended to return bounded page/observation ranges,
but the UI must not present confidence as factual correctness. It may show
`要確認` locations and provide next/previous navigation. It must not rewrite,
delete, accept, or save text automatically.

The current aggregate image confidence value is insufficient for precise
highlighting and must not be stretched into a false per-range claim.

## State Shape

The target state should be small and separate from `EditorTab`:

```ts
type ReferenceDocument =
  | { kind: "pdf"; path: string; name: string; pageCount: number }
  | { kind: "image"; path: string; name: string; url: string; size: number }
  | {
      kind: "text";
      path: string;
      name: string;
      contents: string;
      encoding: TextEncoding;
    };

type ReferenceCompareState = {
  reference: ReferenceDocument;
  origin: "manual" | "import-assist";
  linkedEditorSessionId: string | null;
  followMode: "off" | "following" | "paused";
};
```

This is a design shape, not a requirement to place all state in one component.
Do not add a global store merely for this feature.

## Existing Surfaces To Reuse

- `AppWorkspace` and `editor-preview-grid` for the bounded two-pane shell;
- `PaneResizer` and the existing column-width constraints;
- `ImagePreviewPane` and workspace-contained image loading;
- the current Tab Bar for editable document identity and dirty markers;
- the existing Diff pipeline for an explicit text-to-text `差分を見る` action;
- Import Assist's fixed helper resolution, staging, timeout, and main-window
  command gate;
- existing file-open encoding and size checks for text references.

Reference Compare should add a dedicated `reference` workspace mode rather
than overloading `compare` / Diff state. Diff answers “what text changed?”;
Reference Compare answers “what am I looking at while I edit?”.

## Native / Tauri Contract Direction

The PDF spike should prove a fixed allowlisted helper contract such as:

```json
{"action":"pdf_info","path":"/staged/source.pdf"}
{"action":"render_pdf_page","path":"/staged/source.pdf","page":0,"maxPixels":4000000}
```

Those helper actions remain backend-internal. The frontend-facing Tauri
contract should use an opaque reference handle:

```text
open_pdf_reference(path) -> { referenceId, pageCount, ... }
render_pdf_reference_page(referenceId, pageIndex, viewport) -> bounded image
close_pdf_reference(referenceId)
```

Rust owns the validated source path, any security-scoped access, and any
session-staged container copy for the lifetime of `referenceId`. The frontend
must not receive or replay the staged path. Closing/replacing the reference or
exiting the app releases the access and removes the staged copy.

Expected results contain bounded metadata or page image bytes/data only. The
final naming may differ, but these gates are required:

- main-window-only Tauri commands;
- validated absolute PDF path from explicit user/workspace selection;
- fixed bundled helper path, never a frontend-supplied executable;
- bounded source size, page index, raster dimensions, and response size;
- one active PDF reference handle in the v1.7 MVP, with stale/unknown handles
  rejected;
- no network fetch, embedded script, attachment execution, or shell handoff;
- cancellation or timeout that cannot leave an orphan helper;
- App Store sandbox proof with a user-selected PDF and a workspace PDF.

## Accessibility And Keyboard

- Each pane has a distinct accessible name including role and filename.
- `参照` and `編集` are text labels; role is not conveyed by color alone.
- The separator keeps the existing keyboard resize behavior.
- PDF page changes are announced politely without moving editor focus.
- Page controls, zoom, follow toggle, replace, and close are keyboard
  reachable.
- Reference text is selectable and copyable but exposed as read-only.
- Narrow windows collapse to one visible pane at a time with an explicit
  `参照` / `編集` switch; do not squeeze two unusable columns.

## Failure And Close Behavior

- Import failure: do not open an empty editor or reference pair.
- PDF page render failure: keep the editor intact, show retry/close on the
  reference side, and do not discard the imported draft.
- Reference replacement: explicit action; the Editor Tab remains unchanged.
- Editor Tab close: normal dirty-close protection applies. The reference may
  stay open in the read-only empty-target state.
- App close: normal dirty-close behavior is owned only by Editor Tabs.
- External reference change: show `参照ファイルが変更されました` and offer an
  explicit reload; do not refresh behind the user's comparison position.

## Implementation Slices

### R0 — PDF reference spike

Status: **Source landed (2026-07-10).** Helper actions `pdf_info` /
`render_pdf_page`; Rust session + Tauri commands `open_pdf_reference` /
`render_pdf_reference_page` / `close_pdf_reference`; main-window only; one
active handle; staged path never returned to frontend. Unit tests cover
path/pixel gates and open→render→close with a pinned helper. No complete UI.

Acceptance:

- fixture and live metadata/page results;
- invalid path/type/page rejection;
- timeout/cancel cleanup;
- representative Japanese PDF page renders in a local built app
  *(remaining product smoke; optional before R1)*.

### R1 — Paired shell + text reference

Status: **Source landed (2026-07-10).** Dedicated `ReferenceCompareState`, left
reference / right editor shell, Markdown/text open via workspace context menu,
tab context menu, File menu, and command palette
(`参照ファイルを横に開く…` / `参照として横に開く`). Same-file opens route
to existing buffer-vs-disk review. Narrow windows switch 参照/編集. PDF/image
kinds still placeholder until R2.

Acceptance:

- active Markdown remains the only editable/dirty document;
- reference can be opened, replaced, and closed;
- tab switching keeps the reference pinned without claiming import linkage;
- same-file reference routes to existing disk review;
- narrow-window and keyboard-resizer behavior pass.

### R2 — Image + PDF reference readers

Status: **Source landed (2026-07-10).** Image references use existing
workspace/external image loaders. PDF uses R0 `open_pdf_reference` /
`render_pdf_reference_page` with prev/next page, fit width/page, and
keyboard page navigation. Close/replace releases the PDF handle.
Context menu and reference picker accept text + PDF + image.

Acceptance:

- PDF/image from picker and workspace can open read-only;
- page/zoom controls are keyboard accessible;
- missing/replaced/external-change states preserve the editor;
- App Store sandbox file-access matrix passes
  *(optional packaged smoke; source-covered for open/render/close)*.

### R3 — Import Assist automatic pair + page follow

Status: **Source landed (2026-07-10).** Successful Import Assist opens the
source PDF/image as a linked reference beside the unsaved draft. Link key is
editor `sessionId`. PDF follow uses `<!-- hazakura:import-page index=N -->`
markers under the cursor; manual page change pauses follow; `追従を再開`
resumes. Deleted markers turn follow off without blocking the editor.

Acceptance:

- no auto-save or auto-apply;
- link is keyed by editor `sessionId`;
- deleting markers degrades safely;
- manual PDF navigation pauses follow and explicit resume works;
- Save As preserves the linked editor session without changing source truth.

### R4 — Confidence navigation + reliability

Status: **Source landed (2026-07-10).** Honest **page-level** review only:
OCR paths emit optional `confidence=` on import-page markers; empty pages get
`confidence=0`. UI offers 要確認 prev/next with explicit “目安” copy — never
per-character correctness. Adjacent-page cache (max 3) avoids full PDF preload.
Shell unmount releases the PDF handle. No auto text mutation.

Acceptance:

- confidence is advisory and never framed as correctness;
- no automatic text mutation;
- long PDFs do not preload every page;
- Import Assist cancel, tab close, reference close, and app close leave no
  orphan process or wrong-tab update.

Each slice should be independently reviewable and verified. Do not bundle R0–R4
into one large implementation change.

## Verification Matrix

### Automated

- reference state transitions and `sessionId` linkage;
- same-file routing to existing disk review;
- PDF helper path/type/page/size/window gates;
- page cache eviction and bounded response handling;
- page-marker parsing and deleted-marker fallback;
- zero-based source marker to one-based visible page-label mapping;
- reference close/replace without Editor Tab mutation;
- keyboard navigation and accessible labels;
- no Reference Tab entry in dirty count, backup, save, or restore registries.

### Built app

- workspace Markdown reference + editable Markdown;
- external Markdown reference + editable Markdown;
- image reference + editable Markdown;
- text-layer PDF import → reference + unsaved draft;
- scan PDF OCR → reference + unsaved draft;
- Japanese multi-page PDF page follow;
- manual PDF page change / follow resume;
- Save As, tab switch, dirty close, reference removal, source deletion;
- App Store sandbox picker/workspace access;
- narrow window and keyboard-only operation.

Normal release gates remain:

```bash
npm run build:vite
npm test
cargo fmt --manifest-path src-tauri/Cargo.toml -- --check
cargo test --manifest-path src-tauri/Cargo.toml
npm run build
git diff --check
```

## Explicit Non-goals

- two editable panes or arbitrary split layouts;
- tab-group management, drag-to-split, or multi-window document tiling;
- PDF editing, annotations, forms, attachments, signing, or print-layout tools;
- whole-PDF preload or browser-plugin dependency;
- automatic OCR correction, acceptance, save, or page deletion;
- hidden source metadata database;
- persistent external reference restore in the v1.7 MVP;
- Book Scope, chapter splitting, or multi-file book export;
- cloud OCR, generic AI chat, or Local Assist auto-application;
- Git-aware comparison or merge editing.

## Success Criteria

v1.7 succeeds when a user can:

1. open a PDF, image, or Markdown/text file as a read-only reference;
2. keep one editable Markdown document visible beside it;
3. import a PDF/image and immediately compare the resulting unsaved draft with
   its source;
4. move between imported Markdown page sections and source PDF pages without
   losing editor state;
5. close the reference and return to the ordinary Safe Editor with no hidden
   file mutation or second saved document model.

## Deferred Beyond v1.7

- persistent reference sets or multiple pinned references;
- source annotations and reference bookmarks;
- semantic heading alignment between unrelated Markdown files;
- richer page-region overlays after real confidence-range evidence;
- Book Scope-wide source/reference relationships;
- vertical-writing PDF/OCR review.

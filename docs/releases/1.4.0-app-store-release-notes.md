## Release Position

`1.4.0` is the v1.4 release. It adds two opt-in joke themes (CRT, Shinkai)
as a presentation layer, a review stale-diff warning, and a window-close /
assist-cancel reconciliation, on top of the Observability And Testability
seams. It does not add a new product surface, a second document model,
arbitrary command execution, an external AI/API path, or a broader workspace
model. The joke themes are purely visual overlays; the safe editor surfaces,
Markdown source contract, and export paths are unchanged when they are off.

The product promise is unchanged:

> Markdownで書き、本として読み、ローカルAIで整える。

The core flow remains write -> read -> notice -> return -> revise ->
review -> export. Markdown source remains canonical throughout that flow.

## What Changed In v1.4

### New user-visible features

- **CRT joke theme.** A deliberately hard-to-read "CRT" theme joins the View
  menu and Preferences. It overlays a WebGL CRT shader, scanlines,
  chromatic-aberration text, and a short CRT boot-sequence intro on startup.
  It is an optional joke theme: the safe editor surfaces, Markdown source
  contract, and export paths are unchanged when it is off.
- **Shinkai (deep sea) joke theme.** A second joke theme with its own deep-sea
  shader overlay and boot sequence. Like CRT, it is purely a presentation
  layer that the user opts in to; it does not alter source handling, review,
  or export.
- **Review stale-diff warning.** When a Local Assist proposal is open in
  Change Review and the buffer is edited afterwards, the review view now
  detects the stale diff and shows a warning so the user knows the proposal
  no longer matches the working buffer. Review remains explicit `採用` /
  `破棄`; no auto-save is introduced.
- **Window close confirmation alignment.** Window close while a Local Assist
  window or pending review is active now reconciles with assist cancellation
  so the user is not forced to choose between cancelling assist and closing
  the window. The existing dirty-buffer close confirmation is unchanged.

### Observability And Testability slices

These pure-logic extractions are not user-visible on their own, but they are
the named seams that make the above features safe to ship:

- **E-book reader navigation seams.** The page-commit decision (same-chapter
  floor vs. chapter-change no-floor) and the chapter-cross page-target
  computation were pure functions living privately inside `EBookPane.tsx`.
  They are now exported from `ebookPageTarget.ts` and covered by focused
  tests, so the floor and continuation logic no longer depends on DOM-level
  component tests alone.
- **E-book reader-location seams.** The reader-location assembly,
  location equality, page-to-source-line interpolation, and markdown
  line-counting helpers moved to `ebookReaderLocation.ts` with focused
  coverage for trailing-newline / CRLF / single-line edge cases.
- **PDF export layout tests.** The A4 screen-page-layout calculation (margin
  preset -> content box / column gap in PostScript points) and the point
  value formatter gained fixed-value coverage alongside the existing margin
  and table-annotation tests.
- **EPUB export text-helper seams.** The slugify, YAML-frontmatter stripper,
  XML escaper, document-name title fallback, and image media-type mappers
  moved to `epubTextHelpers.ts` with focused coverage for Japanese-heading
  and frontmatter edge cases.

The Markdown source contract, export output, and App Store lane boundary are
unchanged. The observability slices are pure-logic extractions with new
focused tests; the existing component / integration tests serve as the
regression safety net for the user-visible features.

## Boundary

Hazakura Local Assist remains a preview, availability-gated on-device
writing companion. It has no external AI/API fallback, background
rewriting, auto-save, tool calling, or workspace-wide indexing. Its
proposals remain explicit, unsaved, and Diff / Discard reviewable.

EPUB export remains an explicit single-document workflow. `1.4.0` does not
claim advanced metadata, cover design, navigation editing, multi-file book
management, or a full EPUB production environment. e-book Mode is a reading
preview, not a guarantee that every EPUB reader will reproduce the same
layout.

The App Store lane excludes Agent Workbench, CLI Agent launch, arbitrary
command execution, external AI/API calls, and provider-add UI. The v1.4
observability slices add no new command surface, no DOM editing in Preview,
no second render path, and no telemetry or network reporting.

The CRT and Shinkai joke themes are presentation-only WebGL overlays. They do
not add a network path, a new file format, a new export target, or any
telemetry. They can be turned off from the View menu or Preferences at any
time, and the editor returns to the standard themes without data change.

## Verification Boundary

Local source, build, and audit gates form the candidate evidence for
`1.4.0`. The signed `1.4.0` App Store / TestFlight package has not been
built yet.

- `npm run test`, `cargo test`, `npm run build:vite`, `npm run build`,
  `cargo fmt --check`, and `git diff --check` pass.
- Focused tests cover the four extracted seams (page target, reader
  location, PDF layout, EPUB text helpers). The existing `EBookPane.test.tsx`
  and `epubExport.test.ts` suites pass unchanged, confirming no
  behavior regression.
- CRT theme coverage: `CrtShaderOverlay.test.tsx` and
  `CrtBootSequence.test.tsx`. Shinkai theme coverage: menu / preference
  wiring is exercised via `useAppMenuActionListener.test.tsx`; the Shinkai
  shader and boot-sequence components do not yet have dedicated component
  tests and rely on the shared overlay contract plus manual build-app
  verification.
- Review stale-diff warning coverage: `changeReviewStale.test.ts`.
  Window-close / assist-cancel reconciliation coverage:
  `useAppExitConfirmation.test.tsx` and `appleAssist/AppleAssistWindowApp.test.tsx`.

Building, signing, and uploading the `1.4.0` candidate is a separate
explicitly gated step. Post-release issues enter the observation queue only
when they are reproducible or indicate a hotfix risk.

# v0.13 Distribution Probe

Status: Planning
Scope: App Store / Developer-GitHub distribution readiness before app-quality polish
Authority: Medium
Last reviewed: 2026-06-06

## Decision

v0.13 should focus on **Distribution Probe** rather than Apple Local Assist quality polish.

The immediate goal is to make the app structurally ready for an App Store submission path, or at least to expose the concrete blockers before investing more in UX polish.

This does not mean submitting to the App Store in v0.13. It means proving the lane shape:

- App Store build: Safe Editor + L Mode + Review Desk / Diff + Apple Local Assist, without Agent Workbench or CLI launch.
- Developer / GitHub build: same base app plus optional Agent Workbench, eventually Developer ID signed / notarized.

## Why This Comes Before More Polish

Apple Local Assist quality is likely to improve with OS and model updates. The app should keep the feature available as lightweight local writing help, but v0.13 should not spend most of its budget trying to compensate for weak model output.

Distribution readiness is more structural. If the App Store build cannot cleanly omit Agent Workbench, cannot run the Apple Local Assist helper under sandbox assumptions, or needs a different minimum macOS policy, later UI polish will be built on unstable ground.


## Official Constraints

Apple's current docs establish these constraints:

- Mac App Store apps must enable App Sandbox.
- Notarization requires the Hardened Runtime capability.
- Foundation Models language support should be checked with `SystemLanguageModel.default.supportsLocale()` before generation.
- Foundation Models availability and unavailable reasons must be reflected in UI rather than assumed.
- Temporary App Sandbox exceptions require App Store Connect usage information and should be avoided unless clearly justified.

Sources:

- <https://developer.apple.com/documentation/security/app_sandbox>
- <https://developer.apple.com/help/app-store-connect/reference/app-sandbox-information/>
- <https://developer.apple.com/documentation/xcode/configuring-the-hardened-runtime>
- <https://developer.apple.com/documentation/foundationmodels/support-languages-and-locales-with-foundation-models>

## Probe Findings

### 1. Build Lanes Are Not Yet Separated

Current `tauri.conf.json` has one release lane:

- `bundle.externalBin` always includes `hazakura-apple-assist-helper`.
- Agent Workbench code and UI are part of the same app source.
- The app can hide Agent Workbench by preference, but the App Store lane has not proven build-time omission.

v0.13 should introduce an auditable App Store build profile or equivalent build-time gate.

Minimum acceptable proof:

- App Store build has no Agent Workbench window entry.
- App Store build has no CLI provider UI.
- App Store build cannot call Agent Workbench launch commands, even if frontend code is manipulated.
- App Store build does not bundle xterm / provider-launch surface if practical.

### 2. App Sandbox Is Missing

The local app has no `com.apple.security.app-sandbox` entitlement. That is expected for the current warning-expected / local-app lane, but not acceptable for the Mac App Store lane.

v0.13 should add a draft App Store entitlement profile and decide exact user-selected file access entitlements.

Likely direction:

- `com.apple.security.app-sandbox = true`
- user-selected read/write file access for Markdown/text workspaces
- no arbitrary network entitlement unless Apple Local Assist or app functionality proves a need
- no temporary exceptions by default

### 3. Helper Sidecar Needs Sandbox Proof

Apple Local Assist is currently a bundled Swift sidecar. This is good for keeping Foundation Models out of the cross-platform Rust/React core, but App Store behavior still needs proof.

Open question:

- Can the sandboxed Tauri app spawn the bundled helper through the current `externalBin` path without temporary exceptions or helper-specific entitlements?

v0.13 should test this under a sandboxed local signing profile before changing the architecture.

If the sidecar fails under sandbox:

- first try signing the helper with matching sandbox/hardened-runtime entitlements
- then consider replacing the sidecar with an in-process Swift/AppKit integration only if the sandbox evidence forces it

Do not switch architecture on speculation.

### 4. Minimum macOS Policy Is Still Split

The editor currently declares macOS `11.0`. The Apple Local Assist helper gates Foundation Models at runtime with `#available(macOS 26.0, *)`, `SystemLanguageModel.default.availability`, and `supportsLocale()`.

This is acceptable for a broad editor lane, but App Store messaging must be clear:

- the app can run on older macOS versions
- Apple Local Assist only works where Apple Foundation Models and Apple Intelligence are available

v0.13 should decide whether the App Store build keeps `11.0` or moves to a higher minimum. My recommendation: keep `11.0` for the editor unless App Store packaging or review evidence says otherwise.

### 5. Developer / GitHub Lane Still Needs Developer ID Later

The local app is ad-hoc signed. `spctl` rejection is expected.

This is not a v0.13 blocker if v0.13 is App Store-lane proof. Developer ID signing and notarization can remain a later Developer / GitHub lane task.

## Recommended v0.13 Slices

### Slice 1: Distribution Probe Evidence

- Add and run `npm run probe:macos-distribution`.
- Keep current local-app evidence in this document.
- Verify the script stays read-only.

### Slice 2: App Store Build Gate Design

- Decide how to express build lanes:
  - separate Tauri config
  - environment-gated config generation
  - compile-time Rust cfg + frontend env flag
- Add the smallest no-Agent-Workbench build gate that can be tested.
- Do not remove Agent Workbench from the Developer / GitHub build.

### Slice 3: Sandbox Entitlement Draft

- Add draft entitlements for App Store lane only.
- Keep the current local/developer lane unchanged.
- Confirm local sandbox signing behavior where possible.

### Slice 4: Sandboxed Helper Smoke

- Build the App Store-lane app with Apple Local Assist helper included.
- Verify helper spawn, availability probe, and graceful unavailable states under sandbox.
- If helper spawn fails, record the failure before changing architecture.

### Slice 5: App Review Notes Draft

Prepare a concise App Review note:

- Markdown/text editor.
- User-selected file/folder access.
- Apple Local Assist is optional, on-device, availability-gated, and lightweight.
- AI edits are explicit, unsaved, diff-reviewable, and discardable.
- App Store build has no Agent Workbench, no CLI launch, no arbitrary command execution, no terminal.

## Non-goals For v0.13

- Apple Local Assist model-quality polish.
- Prompt tuning beyond what is needed for review-safe behavior.
- Daily app polish unrelated to distribution readiness.
- TestFlight upload.
- App Store submission.
- Developer ID notarized GitHub release.
- Network LLM fallback.
- New agent/provider features.

## Exit Criteria

v0.13 can be considered successful when:

- App Store build separation is designed and at least partially testable.
- Agent Workbench exclusion has a backend gate, not only hidden UI.
- Sandbox entitlements are drafted and their risks are documented.
- Apple Local Assist helper sandbox behavior is either proven or has a concrete failure report.
- App Review notes are ready as a draft.
- The Developer / GitHub lane remains distinct and does not inherit App Store claims.

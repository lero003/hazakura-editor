# v1.1 Position Continuity Design

Status: Review
Scope: v1.1 daily-use continuity and trust
Last reviewed: 2026-06-29

## Goal

Reader、Editor、Preview を行き来しても、文書ごとの現在位置を失わない。
既知のタブ切替による Editor のカーソル／スクロールリセットを直し、今後
Preview や別の source-preserving surface を追加しても同じ継続モデルへ載せられる
`AppWorkspace` 所有の文書表示状態を作る。

v1.1 の pkg は、位置継続、Preview／リンク遷移の再現確認、Recovery 確認が
まとまった後に作る。画像、Diff 文言、メニュー配置などの独立した polish は
混ぜない。

## Product Boundary

- Markdown/text source は正本のままにする。
- 表示位置はメモリ上の一時状態とし、ファイルへ書き込まない。
- 保存状態、dirty判定、Recovery候補、Diff、AI edit transactionを変えない。
- Git、LSP、terminal、任意コマンド、外部AI/API、project-wide indexingを追加しない。
- undo履歴のタブ別保持、アプリ再起動後の表示位置復元、Book Workspaceは
  v1.1の対象外とする。

## Considered Approaches

### 1. `EditorPane` 内部の文書別Map

変更は最小だが、Reader／Previewから独立した局所状態になり、別surfaceへ
継続モデルを広げるたびに所有ルールが分散するため採用しない。

### 2. `AppWorkspace` 所有の文書表示状態

採用案。`AppWorkspace` が文書キーごとの Editor／Reader／Preview位置を所有し、
各surfaceは状態の報告と復元だけを担当する。現在の `ebookLocations` をこの
registryへ統合し、位置所有ルールを一箇所に寄せる。

### 3. CodeMirrorインスタンスをタブごと保持

undo履歴やplugin stateまで残せるが、非表示Editorの寿命、メモリ、設定反映、
external changeの扱いが複雑になる。位置継続だけを直すv1.1には過大なので
採用しない。

## Architecture

### Document view-state registry

文書表示状態を表す小さな型と更新helperを、Editor固有UIから独立した場所へ置く。
初期shapeは次の責務を持つ。

- `editor`: selection anchor/head と 0..1 の scroll ratio
- `ebook`: chapter/page/source-line からなる既存 `EBookReaderLocation`
- `preview`: 実レイアウトでトップ戻りを再現した場合だけ scroll ratio を使う

registry は `Record<documentKey, DocumentViewState>` とし、一つのsurfaceの更新で
他surfaceの位置を消さない。開いているタブから外れたキーは整理し、長時間利用で
不要な位置情報を増やし続けない。

`documentKey` は現行どおり `activeTab.path || activeTab.id` を使う。Save As／rename時の
完全なrekeyと、再起動を跨ぐ永続化は別のidentity設計が必要なため、今回のpkg条件には
含めない。ただしregistryと更新helperを独立させ、将来stable session idやrekey通知を
追加できる境界にする。

### Editor contract

`EditorPane` は位置の所有者にならない。親から現在文書の初期Editor位置を受け取り、
CodeMirror生成時にselectionを文書長へclampして適用する。scroll ratioはDOM layout後に
復元する。

selection／scrollが変化したら、現在文書の位置を親へ報告する。文書AからBへ切り替える
際はAの最後の状態がregistryに残り、BはB自身の状態または文書先頭を使う。Aへ戻ると
Aの状態を復元する。BへAの状態を継承しない。

既存のPreview同期用 `onScrollRatioChange` は維持する。位置registryへの報告を追加しても、
Editor→Preview同期のタイミングや慣性スクロールguardを変えない。

### Reader contract

既に実装済みのper-document reader locationをregistryの `ebook` fieldへ移す。
明示的なe-book entry、passive remount、`編集に戻る` の既存ルールとテストは維持する。

### Preview and Markdown-link transitions

Previewトップ戻りは、共有scroll containerのidentityが保持されることまでunit testで
確認済みなので、先にbuilt appの実レイアウトで再現する。再現した場合だけ、registryの
`preview`位置をSidePane/AppWorkspace境界で保存・復元する。再現しなければ
`verified no-op`として記録し、推測修正はしない。

ローカルMarkdownリンクは既存の安全な `openPreviewMarkdownLink` → `openWorkspaceFile`
経路を維持する。リンク先を開いて元タブへ戻ったとき、通常のタブ切替と同じEditor位置
contractで復元できることを確認する。外部URLのOS handoffや危険scheme拒否は変更しない。

## Recovery Check

位置継続コードとは分離して、強制終了後のRecoveryを確認する。

1. 専用のローカルfixtureで未保存変更を作り、backup書き込み、再起動、候補表示、明示適用を確認する。
2. Google Driveは専用fixture folderを使い、workspace bookmark復元、backup timing、同一文書再open時の候補抑制を確認する。
3. source lossや候補抑制の不具合を再現した場合だけ、Recoveryを独立したTDD sliceとして直す。
4. Google Driveの専用fixtureを安全に用意できない場合は `manual-blocked` とし、未確認をpkg報告へ明記する。

Recovery確認のためにユーザー文書、cloud credentials、Drive全体を走査しない。

## Testing

TDDで次を固定する。

- Aでカーソル／scrollを動かす → Bへ移動 → Aへ戻るとAの位置へ復元する。
- Bに保存位置がなければAの位置を継承せず先頭から始まる。
- inactive中に文書長が変わってもselection offsetsを有効範囲へclampする。
- L Modeの同一Markdown文書切替は既存どおり不要なremountを起こさない。
- ReaderのA→B→A per-document location regressionを維持する。
- ローカルMarkdownリンク経由のタブ遷移も同じ位置contractを使う。
- Previewは実レイアウト再現結果に応じて、restore testまたは`verified no-op` evidenceを残す。

focused tests後に、repo docsに従ってVite、Rust、App Store surface、distribution、
sandbox-previewのrelease gatesを実行する。manual smokeは実際に操作した範囲だけを
通過として報告する。

## Package Boundary

位置継続の自動テスト、実アプリ確認、Recovery確認、docs同期がまとまった後、現行の
`npm run release:candidate -- --with-app-store-pkg` を使って次の署名済み
App Store／TestFlight pkgを作る。build番号はscriptが現在値から進めた実値を採用し、
事前に固定しない。

pkgについて確認する範囲は、source/build/audit、署名、entitlements、checksum、
distribution probe、sandbox-previewまでとする。App Store Connect upload、TestFlight配布、
tag、push、公開はこの依頼の対象外とする。

## Completion Criteria

- 文書A→B→AでEditorのカーソルとscrollが戻り、Bへ状態が漏れない。
- 既存Reader位置継続とEditor↔Reader bridgeが退行しない。
- Preview／リンク遷移は再現結果がfixまたは`verified no-op`として記録される。
- Recoveryのlocal結果とGoogle Drive結果、または明示的な`manual-blocked`が記録される。
- current-work、current-status、handoff、follow-up、smoke checklistが実状態と一致する。
- release gatesを通過した署名pkgと、そのlocal provenanceが生成される。
- upload、tag、push、公開は行わない。

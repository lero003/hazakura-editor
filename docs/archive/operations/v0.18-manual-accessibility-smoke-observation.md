## Why Archived

The current slice was a code-and-observation pass. VoiceOver and
Increase Contrast need the user's actual accessibility settings on
a real Mac, so the live observation items are still pending.
Putting the detailed DOM / CSS / focus-trap walk-through into the
operational smoke checklist would bloat it without adding anything
a future user-side observation can rely on. Keep the checklist
shallow; keep the depth here.

## 1. Help Readability — Code-Level Detail

### Surfaces Confirmed

- `src/components/app/helpDocs/index.ts` exports five separate Help
  documents wired through the `helpDocsByMode` map: Privacy Policy,
  Local Data Disclosure, Support Diagnostics, About, Open Source
  Acknowledgements.
- All five documents share the `PrivacyPreferencesPane` shell with
  a kicker, boundary card, scrollable body, and footer note.
- H1 title is suppressed (`.sr-only` in `PreferencesDialog`) for
  Help modes; the article uses `aria-label={doc.title}` so the
  visible Help body still has a programmatic name.

### Anchor Integrity

- `injectHelpDocSectionAnchors` rewrites H2 tags to carry the
  section table's `testId` so a future "copy link to section"
  affordance can rely on stable `id`s.
- `src/components/app/PrivacyPreferencesPane.test.tsx` pins the
  section table ↔ H2 1:1 correspondence and asserts the rendered
  H2 carries the `testId`.

### Scroll

- Body lives in `.privacy-tab-panel-scroll`; long documents
  (e.g. `local-data-disclosure.md`) need a user-side live
  observation to confirm the scroll affordance.

### Narrow Widths

- Dialog has a max-width; responsive narrow-width behaviour is
  CSS-only and left for the user-side observation.

### Observations (Recorded, Not Fixed)

- Copy overlap across Privacy Policy and Local Data Disclosure
  ("Files you choose", "Network and analytics") is intentional
  scope for the existing P2 slice `Help copy overlap cleanup`
  in `docs/current-work.md`. This smoke did not restructure
  Help copy.
- Help documents avoid hand-written external links per the
  `helpDocs/index.ts` header note (the preview's allow-list
  permits `http:`, `https:`, `mailto:`, `tel:` but Help
  documents are routed through the same preview sanitizer
  and intentionally use no external links).

## 2. Full Keyboard-Only Traversal — Code-Level Detail

### Dialogs

- `PreferencesDialog`:
  `preferencesCloseButtonRef` is wired through
  `useDialogInitialFocus`, and `Tab` / `Shift+Tab` is trapped
  inside the dialog by
  `trapFocusInElement(preferencesDialogRef.current, event)`
  (see `src/lib/focusTrap.ts`). `Esc` is handled by
  `useModalKeyboardGuard`.
- `DirtyTabCloseDialog` and `AppCloseDialog` (in
  `src/components/app/CloseDialogs.tsx`): both have a
  `cancelButtonRef` and a `dialogRef`; initial focus lands
  on the cancel button and `Tab` is trapped via
  `useModalKeyboardGuard`. `Esc` cancels.
- `MoveToTrashConfirmDialog` (in
  `src/components/app/MoveToTrashConfirmDialog.tsx`): does
  NOT register a dialog ref, an initial-focus target, or a
  focus-trap path through `useModalKeyboardGuard`. The
  visible buttons still work with mouse and `Tab` reaches
  them in DOM order, but the dialog does not focus-trap
  and does not set initial focus. Recorded as a separate
  P2 follow-up in `docs/current-work.md`; no fix attempted
  in this slice.

### TabBar

- Text tabs and the image preview tab are all in one
  `role="tablist"`.
- `ArrowLeft` / `ArrowRight` / `Home` / `End` move focus
  along the merged order with no wrap (boundary clamp is
  intentional, mirrors WAI-ARIA "no wrap" guidance).
- The close button is intentionally outside the arrow-key
  traversal so the user can still Tab past it. VoiceOver
  users will still reach the close button via quick-nav.

### Workspace Sidebar

- `+` button has `aria-haspopup="menu"` with a controlled
  `New` popover; the popover closes on `Esc` and on
  outside click (`mousedown` listener registered when
  the menu opens in `src/components/workspace/WorkspaceSidebar.tsx`).
- Tab order through the tree is driven by the rendered
  buttons.

### Sidebar Collapse / Restore

- In normal mode the sidebar exposes a `PanelLeftClose`
  collapse button (`AppWorkspace.tsx`).
- When collapsed, a `workspace-sidebar-rail` shows a
  `PanelLeftOpen` restore button with
  `aria-label={copy.restoreWorkspaceSidebar}`.
- The rail is intentionally hidden in L Mode (L Mode keeps
  its own temporary workspace drawer).

### Status Bar

- `role="status"` + `aria-live="polite"` pinned by
  `StatusBar.test.tsx`; format selects are absent in L Mode
  but `role="status"` remains so status messages still reach
  assistive tech.
- Encoding / line-ending selects have explicit
  `aria-label={lineEndingAriaLabel}` /
  `aria-label={encodingAriaLabel}` for VoiceOver.

### Localized Copy

- Close dialog copy and Move-to-Trash dialog copy are
  localized for Japanese, hiragana, and English; dialog
  titles are also localized. No copy change attempted in
  this slice.

## 3. VoiceOver Tab-Bar Announcement — Code-Level Detail

### Text Tab

- `role="tab"` with `aria-selected={tab.id === activeTabId}`.
- When the shared `isDirty()` returns true, the tab button
  gets `aria-describedby="tab-dirty-{id}"` and a hidden
  `<span id="...">unsaved</span>` lives next to the
  decorative `.tab-dirty-dot` (`aria-hidden="true"`).
- `src/components/app/AppTopChrome.test.tsx` pins
  `description: "unsaved"` for both contents-dirty and
  encoding-only-dirty tabs. The encoding-only dirty slice
  in `src/features/editor/editorTabs.ts` did not regress
  the dirty description.

### Image Preview Tab

- Same `role="tablist"`. `aria-selected` is hard-`true`
  (the slot is a display-only companion).
- Arrow keys still reach it so VoiceOver users can
  navigate to the close button next to it.

### Close Button

- `aria-label="Close {tab.name}"` (text tab and image
  preview tab both). See `src/components/editor/TabBar.tsx`.

## 4. Increase Contrast — Code-Level Detail

`src/styles/a11y.css` activates the following rules under
`@media (prefers-contrast: more)`, the macOS-reported media
feature when Increase Contrast is on:

- Strengthens `--border`, `--border-strong`,
  `--selection-bg`, `--cm-selection-bg`, and
  `--scrollbar-thumb*` so theme tokens stay legible against
  the surface.
- Interactive `:focus-visible` outline widens from
  `2px / offset 2px` to `3px / offset 1px`.
- Theme-graded focus rings (yakou / shokou) widen from 4px
  to 5px.
- `.tab-item`, `.tab-button`, `.close-dialog`,
  `.preferences-dialog`, button / select / input /
  textarea borders widen from 1px to 2px.
- `.status-bar` gains a 2px top border.
- `.save-affirmation` gains a 1.5px outer ring (added via
  `box-shadow` so the 16px round badge stays legible).
- Placeholder text and scrollbar thumbs get a stronger
  color mix so they stop disappearing.

The `.tab-dirty-dot` uses `background-color: var(--accent)`
in `src/styles/editor.css`, which the strengthened theme
tokens make legible in Increase Contrast; the dot is also
purely decorative (`aria-hidden="true"`) and does not
need a higher-contrast version of itself.

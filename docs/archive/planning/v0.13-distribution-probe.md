# v0.13 Distribution Probe

Status: Tagged checkpoint
Scope: v0.13.0 App Store / Developer-GitHub distribution readiness probe and remaining proof
Authority: Medium
Last reviewed: 2026-06-06

## Decision

v0.13 focused on **Distribution Probe** rather than Apple Local Assist quality polish.

The immediate goal is to make the app structurally ready for an App Store submission path, or at least to expose the concrete blockers before investing more in UX polish.

This did not mean submitting to the App Store in v0.13. It meant proving the lane shape:

- App Store build: Safe Editor + L Mode + Diff / explicit change review + Apple Local Assist, without Agent Workbench or CLI launch.
- Developer / GitHub build: same base app plus optional Agent Workbench, eventually Developer ID signed / notarized.

## Why This Came Before More Polish

Apple Local Assist quality is likely to improve with OS and model updates. The app should keep the feature available as lightweight local writing help, but v0.13 did not spend most of its budget trying to compensate for weak model output.

Distribution readiness is more structural. If the App Store build cannot cleanly omit Agent Workbench, cannot run the Apple Local Assist helper under sandbox assumptions, or needs a different minimum macOS policy, later UI polish will be built on unstable ground.

## Official Constraints

Apple's current docs establish these constraints:

- Mac App Store apps must enable App Sandbox.
- Notarization requires the Hardened Runtime capability.
- Foundation Models language support should be checked with `SystemLanguageModel.default.supportsLocale()` before generation.
- Foundation Models availability and unavailable reasons must be reflected in UI rather than assumed.
- Temporary App Sandbox exceptions require App Store Connect usage information and should be avoided unless clearly justified.

Sources:

- <https://developer.apple.com/documentation/security/app_sandbox>
- <https://developer.apple.com/help/app-store-connect/reference/app-sandbox-information/>
- <https://developer.apple.com/documentation/xcode/configuring-the-hardened-runtime>
- <https://developer.apple.com/documentation/foundationmodels/support-languages-and-locales-with-foundation-models>

## Probe Findings

### 1. Build Lanes Are Partially Gated

Current `tauri.conf.json` still has one bundle config:

- `bundle.externalBin` always includes `hazakura-apple-assist-helper`.
- Agent Workbench code is still part of the same app source.
- The first App Store-lane gate now uses `HAZAKURA_DISTRIBUTION_LANE=app-store` / `VITE_HAZAKURA_DISTRIBUTION_LANE=app-store`.
- In that lane, frontend preferences normalize `external-cli` to `apple-local`, hide the CLI Agent setting, and keep Agent Workbench inactive.
- Rust command wrappers reject Agent Workbench IPC when the App Store distribution lane is compiled in.

This is an initial backend/UI gate, not a complete App Store bundle profile.

Minimum acceptable proof:

- App Store build has no CLI provider UI. Initial gate: done.
- App Store build cannot call Agent Workbench launch commands, even if frontend code is manipulated. Initial gate: done for Agent Workbench command wrappers and Agent window open.
- App Store build has no Agent Workbench window entry. Initial gate: native menu / command palette Agent entries are omitted in the `app-store` lane, Agent-only IPC is skipped or rejected, and `agent.html` is omitted from the Vite App Store build input.
- App Store build does not bundle xterm / provider-launch surface if practical. Partially done: the App Store Vite build no longer emits `agent.html`; dependency/package-level pruning and capability-profile separation remain future hardening, not current submission proof.

Use this preview command while the lane is being shaped:

```bash
npm run build
```

`npm run build` is intentionally mapped to the normal App Store preview lane so the normal app bundle does not show Developer / GitHub lane markers. `npm run build:app-store-preview` is the explicit alias for the same lane. These commands are structural previews only. They do not imply App Store signing, sandbox entitlements, notarization, upload, or review readiness.

For local release-lane inspection, build both macOS lanes side by side:

```bash
npm run build:macos-lanes
```

This leaves the App Store preview lane at the normal app path and copies the Developer / GitHub lane to a separate Dev-named bundle:

- `src-tauri/target/release/bundle/macos/hazakura editor.app` = App Store preview / normal lane
- `src-tauri/target/release/bundle/macos/hazakura editor Dev.app` = Developer / GitHub lane

The Dev bundle is re-identified after copy:

- `CFBundleName` / `CFBundleDisplayName` = `hazakura editor Dev`
- `CFBundleIdentifier` = `lab.hazakura.note.dev`
- ad-hoc re-signed with the hardened runtime flag

This keeps the local artifact name, macOS app identity, and warning-expected DMG lane distinct from the App Store preview lane.

The Dev lane also renders a small `DEV` badge in the app chrome and uses `hazakura editor Dev` for the main window title. The App Store preview lane does not render the badge.

The warning-expected GitHub DMG preview now packages the Dev lane bundle:

```bash
npm run build:dmg-preview
```

This produces a `hazakura-editor-dev_...-warning-expected.dmg` artifact and keeps the normal `.app` name reserved for the App Store preview lane.

### 2. App Sandbox Is Missing

The local app has no `com.apple.security.app-sandbox` entitlement in the normal developer build. That is expected for the current warning-expected / local-app lane, but not acceptable for the Mac App Store lane.

v0.13 now has draft App Store entitlement profiles:

- `src-tauri/entitlements/app-store-app.plist`
- `src-tauri/entitlements/app-store-helper.plist`

Current draft direction:

- `com.apple.security.app-sandbox = true`
- user-selected read/write file access for Markdown/text workspaces in the app bundle
- app-scope bookmarks for future persistent workspace/recent-file access
- no arbitrary network entitlement unless Apple Local Assist or app functionality proves a need
- no temporary exceptions by default

Use this diagnostic command to copy the built app to a temp directory, ad-hoc re-sign it with the draft entitlements, and probe the bundled helper:

```bash
npm run smoke:macos-sandbox-preview
```

Current result on 2026-06-06: **the app and helper can be ad-hoc signed with sandbox entitlements and `codesign --verify --deep --strict --verbose=2` passes. The real sandbox-signed `hazakura-editor` parent can spawn the bundled helper with `com.apple.security.inherit`, and the helper returns an availability envelope.** The smoke also keeps a representative sandboxed parent app check as a narrow independent control.

Important nuance: direct shell execution of a sandbox-signed helper still fails with exit status `133` / `SIGTRAP`. Crash reports include `libsystem_secinit` frames and signatures such as `Process is not in an inherited sandbox.` and `Unable to get bundle identifier ... Info.plist ... has no value for kCFBundleIdentifierKey.` That direct-helper failure is now treated as expected for the inherited-sandbox helper shape, not as proof that the sidecar architecture is unusable.

Interpretation: the sidecar approach is still viable. The remaining proof is moving from diagnostic ad-hoc sandbox signing to the actual App Store signing / provisioning path once Apple Developer account access is available.

### 3. Helper Sidecar Needs Sandbox Proof

Apple Local Assist is currently a bundled Swift sidecar. This is good for keeping Foundation Models out of the cross-platform Rust/React core, but App Store behavior still needs proof.

Open question:

- Can the sandboxed Tauri app spawn the bundled helper through the current `externalBin` path without temporary exceptions or helper-specific entitlements?

Initial direct helper execution failed after sandbox signing, but inherited-sandbox parent-spawn smoke passed through the real `hazakura-editor` parent. The remaining question is now distribution-account-specific:

- Can the same shape pass Apple Developer / App Store signing, provisioning, and upload validation?

If the sidecar fails under sandbox:

- first try signing the helper with matching sandbox/hardened-runtime entitlements
- then test parent-spawn behavior from a sandboxed app, not only direct shell execution
- then consider replacing the sidecar with an in-process Swift/AppKit integration only if the sandbox evidence forces it

Do not switch architecture on speculation.

### 4. Minimum macOS Policy Is Still Split

The editor currently declares macOS `11.0`. The Apple Local Assist helper gates Foundation Models at runtime with `#available(macOS 26.0, *)`, `SystemLanguageModel.default.availability`, and `supportsLocale()`.

This is acceptable for a broad editor lane, but App Store messaging must be clear:

- the app can run on older macOS versions
- Apple Local Assist only works where Apple Foundation Models and Apple Intelligence are available

v0.13 kept the editor-wide `11.0` minimum and runtime-gated Apple Local Assist. The recommendation remains: keep `11.0` for the editor unless App Store packaging or review evidence says otherwise.

### 5. Developer / GitHub Lane Still Needs Developer ID Later

The local app is ad-hoc signed. `spctl` rejection is expected.

This is not a v0.13 blocker because v0.13 is an App Store-lane proof checkpoint. Developer ID signing and notarization remain a later Developer / GitHub lane task.

## Non-goals For v0.13

- TestFlight upload.

## Exit Criteria

v0.13 is considered successful because:

- App Store build separation is designed and testable.
- Agent Workbench exclusion has backend gates, not only hidden UI.
- Sandbox entitlements are drafted and their risks are documented.
- Apple Local Assist helper sandbox behavior is proven through the sandboxed parent path, with the remaining Apple Developer / App Store signing proof called out separately.
- App Review note shape is ready as a draft.
- The Developer / GitHub lane remains distinct and does not inherit App Store claims.

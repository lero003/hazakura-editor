# v0.27 Refinement Slice Plan

Status: Completed archive
Scope: Historical v0.27 post-v0.26 refinement execution phases
Authority: Medium
Last reviewed: 2026-06-21 (archived after v0.28 docs cleanup)

## Summary

v0.27 は **新しい面を増やすのではなく、既存の編集空間を"ひとつの連
続した安全な編集空間"に引き締める** フェーズとする。

post-v0.26 の時点で、no-workspace 執筆、e-book 空状態、EPUB export
beta は揃った。次に価値が出るのは、読む・書く・比べる・保存するが
現状よりも滑らかにつながる体験を整えることである。

この文書は `docs/post-v0.25-product-refinement-plan.md`（lens）を
v0.27 の実行フェーズに落とした構成メモである。lens を置き換えるも
のではなく、current-work.md / roadmap.md と合わせて使う。

North star は変わらず:

> AIが書いたMarkdownを、本として読み、差分で直す。

v0.27 はこの north star に向かう前の **足場整備** であり、AI 取り込
み（`docs/ai-markdown-ingest-plan.md`）は v0.28 以降の候補として置く。
v0.27 の間に app shell・mode 制御・大容量挙動を安定させることで、重
い review / import workflow をより安定した基盤の上で開ける。

## Refinement Lens（全スライス共通の判断基準）

各スライスはこの問いを通す:

> これは読む・書く・比べる・保存するを、より"ひとつの連続した安全な
> 編集空間"に感じさせるか?

当てはまらない small polish は、v0.27 の軸から外れるので別キューに
回す。

## Product Boundary

v0.27 は既存の Safe Editor 境界を守る:

- Markdown / text source は引き続き canonical
- Git / LSP / terminal / plugin / 任意コマンド実行 / project-wide
  indexing / auto-apply / auto-commit は入れない
- Preview DOM 編集、`contenteditable`、第2の saved document model
  は作らない
- 新しい mode も増やさない（既存 Normal / L Mode / e-book / Preview
  / Diff の連続性を整える）

AI 取り込み・Workspace As Book の情報アーキテクチャ変更・Native
Vibrancy Phase 2（macOS 26 floor）は v0.27 の対象外。それぞれ独立
lane として別途判断する。

## Phase 構成

各 Phase は小さく、検証可能で、source-preserving に保つ。

| Phase | スライス | lens の対応 | 想定リスク |
|---|---|---|---|
| 1 | 大容量文書の初期描画安定化 | Performance And Reliability | 中 |
| 2 | One Editing Space 最小: モード切替コンテキスト保持 | One Editing Space / Flow-Preserving | 小 |
| 3 | heading jump 即時化 + 編集位置履歴 | Flow-Preserving Editing | 小 |
| 4 | status bar 構造クリーンアップ | UI that recedes | 小 |

Phase の順序は推奨順であり、各 Phase は独立して切れる。ただし Phase
1（大容量描画）は現状挙動の調査が前提で、結果によっては `verified
no-op` で閉じる可能性もある。

## Phase 1: 大容量文書の初期描画安定化

現状の観察（ユーザー報告、未コード確認）:

- 大容量ファイルを開いたとき、最初の描画が多少乱れることがある

これは post-v0.25 lens の「Performance And Reliability」に直結し、
かつ "trust work" に位置づけられている。v0.27 で最初に調べる価値が
高い。

リスク仮説（要確認、推測で直さない）:

- CodeMirror 6 の初期化と document 読み込みのタイミング
- Preview / e-book Mode の初回レンダリングが同期的に走る
- e-book Mode の CSS Columns ページ計算が未描画の状態で走る
- 画像 inlining の非同期解決前のレイアウト揺れ

受け入れ基準:

- 再現条件を特定し、focused regression / smoke で証明できる
- 明らかな初期描画乱れを取り除く（段階レンダリング、初回レイアウト
  の遅延、または該当 mode の表示安定化）
- 通常の開く・編集・保存・preview・e-book 動作を壊さない

終了状態は以下のいずれか:

- `implemented`: 再現と修正が確認済み
- `verified no-op`: 調査の結果、安全な小修正が見当たらない

manual app smoke が必要な項目（大容量ファイル実機確認）は
`manual-blocked` として残してもよい。

## Phase 2: One Editing Space 最小

目的: L Mode / e-book / Preview / Diff を切り替えたとき、同じ文書の
別レイヤーを通っているように感じさせる。

現状は未調査。スクロール位置・選択範囲・heading context が既に保持
されていれば `verified no-op` で閉じる。

候補（調査結果で絞る）:

- モード切替後もスクロール位置・カーソル位置を保持
- 現在 heading / 節 context を切替越しに維持
- 切替自体の視覚的な「離脱感」を減らす（過剰な再描画抑制など）

受け入れ基準:

- 少なくとも1つの再現する「離脱ポイント」を特定して修正
- 既存の mode 切替・dirty 状態・save 衝突回復を壊さない

## Phase 3: Flow-Preserving Editing

目的: 書き手の思考の流れを守る小さな操作を整える。

候補:

- heading jump を即時・予測可能に
- 直近の編集位置に戻れる軽量な編集位置履歴（永続化しなくてよい）
- focus-paragraph 的な表示は v0.27 では入れない（別 spike）

受け入れ基準:

- heading jump の遅延・外れを再現して修正、または `verified no-op`
- 編集位置履歴は session 内のみで、裏書き・indexing なし

## Phase 4: status bar 構造クリーンアップ

現状の `statusDetail` 1本の文字列は v0.20 の stopgap と明記されてい
る（current-work.md / roadmap.md の v0.21+ 項目）。これを v0.27 の
"UI that recedes" として統合する。

受け入れ基準:

- 優先度ごとの metadata field に分割
- line-ending / encoding 制御は常時到達可能を維持
- final-newline / 行列・選択・heading context は hover / popover /
  adaptive secondary に退避
- L Mode の既存挙動を壊さない

## Out Of Scope（v0.27 対象外）

- AI Markdown 取り込み（`docs/ai-markdown-ingest-plan.md`）→ v0.28 候補
- Workspace As Book の情報アーキテクチャ変更 → 別 lane
- 複数ファイル EPUB / 縦書き / 高度メタデータ → 別 lane
- Native Vibrancy Phase 2（window-vibrancy + macOS 26 floor）→
  App Store lane 決定を伴う独立 lane
- Plugin / provider / terminal / Git / LSP / 任意コマンド実行
- Preview DOM 編集 / `contenteditable` / 第2 saved document model

## Verification Direction

各 Phase は以下を証明する:

- 変更が既存の開く・保存・preview・e-book・Diff 動作を壊さない
- focused regression / smoke で効果を証明、または `verified no-op`
- manual app smoke が必要な項目は `manual-blocked` として明示
- `npm run test`, `npm run build:vite`, `git diff --check` を基本 gate とする

## How To Use This Memo

1. `docs/current-work.md` を起点にする
2. この文書から Phase を1つ選ぶ（推奨順: Phase 1 → 4）
3. 実装前に証明経路を特定する（regression test / source 調査 /
   built app smoke / manual App Store / TestFlight 証拠）
4. スライスが lens の方向性を変える場合のみ、この文書を更新する
5. 各 Phase は `implemented` / `manual-blocked` / `verified no-op`
   のいずれかで閉じる

この文書が成功したのは、プロダクトを重くせず、ワンランク完成度を上
げたと感じられるときである。

# v1.11 OKF Draft Compatibility Preview

Status: Accepted design; S0–S3 source landed; S4 distribution gate open
Scope: OKF v0.1 Draft の明示的・読み取り中心の互換確認と配布品質ゲート
Authority: High for v1.11 implementation
Last reviewed: 2026-07-15 (UX progressive disclosure: copy + result hierarchy)

## Decision

v1.10 Single-document Structure Foundation は、source 実装と代表的な packaged
smoke を完了した。残る横断的な手動確認は、v1.11 の distribution-confidence
matrix に引き継ぐ。

v1.11 では、OKF (Open Knowledge Format) v0.1 Draft を対象に、ユーザーが
明示的に選んだ workspace / subfolder を一度だけ確認する **OKF Draft
Compatibility Preview** を実装する。これは OKF bundle の読み取り、互換性の
助言、既存 Markdown tab への移動を行う preview であり、Book Scope、章順管理、
multi-file edit、manifest authoring ではない。

App Store への次の公開は、この preview と v1.11 distribution-confidence gate
の後に別途判断する。現在の in-tree version `1.10.0` を、この docs-only slice で
変更したり、upload / submit / publish したりしない。

## Specification Snapshot

Implementation contract is pinned to the official **Version 0.1 — Draft**
snapshot reviewed on 2026-07-15:

- commit: [`ee67a5c`](https://github.com/GoogleCloudPlatform/knowledge-catalog/blob/ee67a5c/okf/SPEC.md)
- raw source: [`SPEC.md`](https://raw.githubusercontent.com/GoogleCloudPlatform/knowledge-catalog/ee67a5c/okf/SPEC.md)

Do not silently follow `main`. A later upstream revision requires a new design
review and fixture update. Do not vendor the complete upstream specification.

The base specification defines a knowledge bundle, not a book or publishing
format. Hazakura describes this surface as **OKF v0.1 Draft compatibility**,
not complete OKF support or an OKF-defined book workflow. Book order, chapter
semantics, whole-book reading, and export remain Hazakura product decisions for
v2.

The upstream draft contains an apparent tension: its reserved-file section says
`index.md` has no frontmatter, while its versioning section permits
`okf_version` in the bundle-root `index.md`. The local consumer contract below
resolves that tension permissively instead of turning it into a hard rejection.

## User-visible Contract

1. The user opens a workspace as today.
2. From the main Safe Editor window, the user explicitly invokes
   **`知識フォルダ（OKF）を点検`** (Command Palette / folder context;
   English: **Review knowledge folder (OKF)**) for the workspace root or a
   selected directory. Opening a workspace alone never starts OKF analysis.
3. When a directory is selected, it is the bundle root. Otherwise the workspace
   root is used. With no open workspace, the action is disabled with an
   explanatory status.
4. Hazakura performs one bounded, cancellable disk-snapshot read inside that
   root and creates an in-memory compatibility result.
5. A lightweight, read-only review surface shows a plain-language status
   (including ordinary-Markdown-folder framing), priority findings, optional
   advice, the Markdown file list, and an action to open a concept in the
   existing Markdown editor tab. Spec pin and raw counts live under a details
   disclosure, not as the primary reading path.
6. Dismissing the surface discards the derived result. Rerunning replaces it
   only after a usable new result exists.

The surface is reachable from the selected directory context and Command
Palette. It does not add a permanent top-level mode or toolbar group. It is not
available as an Agent Workbench or Assist command.

The surface states that results reflect files on disk. An open dirty tab may
differ from the scan. Opening a listed concept reuses an already-open dirty tab
instead of replacing its buffer with disk content.

The review surface renders metadata and lists only. It does not render Markdown
HTML, inline HTML, or Markdown images.

## UX Progressive Disclosure

v1.11 keeps quality checking explicit and does not treat every workspace as OKF.

**Must not:**

- run full OKF discovery / YAML / link analysis on workspace open or app start;
- background re-scan, file watcher, or persistent OKF index / cache database;
- auto-repair frontmatter, `index.md`, links, or types;
- claim formal OKF certification (results are Draft compatibility guidance only).

**May (later polish, still explicit or zero-content-read):**

- rename and reframe copy so writers understand the action without knowing OKF;
- classify a completed result as `okf-like` vs ordinary Markdown folder for status
  wording only (pure UI heuristic over an existing `OkfReviewResult`);
- show a one-line sidebar chip when the already-loaded workspace tree shows a
  root `index.md`, if and only if clicking it starts the same explicit review
  and no Markdown body is read for the hint;
- File menu entry and in-panel help / static minimal examples.

Ordinary Markdown folders are a normal Hazakura use case. High failure counts on
files without `type` frontmatter must not be presented as “the workspace is
broken”; they should explain that OKF concept rules apply only when treating the
folder as a knowledge bundle.

## Compatibility Status Matrix

`index.md` and `log.md` are classified by reserved filename first. Frontmatter,
including `type:`, never turns either file into a concept.

| Input | Classification | Result |
|---|---|---|
| non-reserved `.md` without leading frontmatter | concept | compatibility failure |
| non-reserved `.md` with unclosed or unparseable YAML | concept | compatibility failure |
| concept with missing, empty, or non-string `type` | concept | compatibility failure |
| concept with unknown `type` or producer-defined keys | concept | accepted; optional advice only |
| root `index.md` with `okf_version: "0.1"` | reserved index | accepted |
| root `index.md` with other keys or unknown version | reserved index | best-effort advice; never a concept |
| nested `index.md` with frontmatter | reserved index | advice; never a hard failure or concept |
| absent, empty, heading-less, or list-less `index.md` | reserved/absent index | advice only |
| `log.md` without valid `## YYYY-MM-DD` entries or with entries out of order | reserved log | advice only |
| unreadable or non-UTF-8 Markdown | unreadable file | visible unreadable status; not parsed as YAML |
| broken or out-of-scope local link | relationship advice | advice only |

Optional metadata such as `title`, `description`, `resource`, `tags`, and
`timestamp` never becomes a hard requirement. Missing `index.md`, unknown
versions, and broken links remain tolerable. The preview is a compatibility
check, not a producer certification tool.

The preview never rewrites frontmatter or links. Future explicit fixes, if
approved, must be ordinary reviewable editor transactions rather than automatic
bundle repair.

## YAML Parsing Contract And Dependency Gate

- Reuse `findYamlFrontmatter` only to detect the leading frontmatter range. It
  is not a YAML parser and must not be presented as one.
- S1 owns YAML interpretation in the side-effect-free TypeScript model via the
  direct runtime dependency `yaml` (declared as `yaml@^2.9.0` in
  `package.json`; the exact resolved version is fixed by `package-lock.json`).
  A valid concept requires parseable YAML mapping and a non-empty string
  `type`; numeric, sequence, object, null, and whitespace-only values fail.
- Unclosed delimiters and parser errors are compatibility failures. Nested
  mappings, anchors, and aliases are accepted when the `yaml` parser accepts
  them. Value conversion must be cycle- and depth-safe so unknown fields never
  throw; v1.11 does not map them into the editor's existing frontmatter model.
- Unknown keys only need safe display/classification. v1.11 neither rewrites
  nor round-trips parsed YAML.
- Do not reintroduce a hand-rolled subset parser for conformance decisions.

## Markdown Link Resolution Contract

The initial preview recognizes standard inline links `[text](destination)` in
the Markdown body of concepts, `index.md`, and `log.md`. It does not scan
frontmatter values such as `resource`. Reference-style link definitions may be
added only in a later reviewed slice.

For target-existence advice:

1. Decode and normalize the local destination safely, then ignore query and
   fragment components for existence checks.
2. `http:`, `https:`, `mailto:`, and `tel:` are external relationships. Show
   them as external; do not fetch them and do not report them as broken.
3. `/foo.md` is relative to the selected OKF bundle root, not the filesystem
   root or workspace root above it.
4. `./x.md` and `../y.md` are relative to the source file's directory.
5. A normalized target outside the selected root is `out-of-scope` advice. Do
   not read it or offer an in-app open action.
6. Extensionless and trailing-slash targets check the normalized file or
   directory and, for a directory, its `index.md` convention. They do not cause
   an unbounded secondary walk.
7. Other schemes, malformed destinations, and absolute filesystem paths are
   unsupported advice and never an OS handoff from the review surface.

Opening a recognized internal Markdown target still uses the existing Safe
Editor containment and tab-open path. Link recognition does not grant new file
access.

## Discovery Contract And Budgets

S2 scans disk state only and counts walk entries separately from Markdown files
and Markdown bytes. Only regular files whose extension ASCII-case-insensitively
equals `.md` are candidates. Non-Markdown files are ignored. Symlinked files
and directories are never followed.

The first implementation pins these constants in Rust with focused boundary
tests:

| Constant | Initial value | Meaning |
|---|---:|---|
| `MAX_OKF_WALK_ENTRIES` | `2_000` | all inspected directory entries |
| `MAX_OKF_MARKDOWN_FILES` | `200` | candidate Markdown files |
| `MAX_OKF_FILE_BYTES` | `10 * 1024 * 1024` | one Markdown file |
| `MAX_OKF_TOTAL_BYTES` | `32 * 1024 * 1024` | total Markdown bytes read |
| `MAX_OKF_DEPTH` | `16` | relative directory depth |

These are OKF-specific contracts, even where they align with existing
`MAX_WORKSPACE_ENTRIES = 2000` or `MAX_EDITABLE_BYTES = 10 MiB`. Discovery may
reuse the existing deterministic workspace-search/path guards, but must not
inherit `MAX_WORKSPACE_SEARCH_FILES = 500` implicitly.

Results are thin and serializable:

```ts
type OkfReviewResult = {
  summary: OkfReviewSummary;
  files: OkfReviewFile[];
  findings: OkfReviewFinding[];
  findingsTruncated: boolean;
  scannedEntries: number;
  scannedMarkdownFiles: number;
  totalBytesRead: number;
  truncated: boolean;
  truncationReason?: OkfTruncationReason;
  cancelled: boolean;
  source: "disk";
};
```

`findings` is a flat list keyed to files/locations; do not build or persist a
graph adjacency model. Stable relative-path ordering is required. Model work
and rendered rows are bounded independently of discovery: scan at most
`1_000_000` body characters and `500` inline links per file, retain at most
`1_000` findings, apply a linear parser-work budget to malformed link/code
runs, and set `findingsTruncated` when analysis omits the rest.

The first bounded or cancelled run returns its safe partial result with
`truncated` / `cancelled` state. If a later rerun fails before producing a safe
result, keep the prior usable result visible and show the rerun failure. An
unreadable or over-budget file receives a visible per-file status and is not
partially parsed.

## Layer Responsibilities

| Slice / layer | Owns | Must not own |
|---|---|---|
| S1 TypeScript pure model | frontmatter interpretation, concept/reserved classification, inline-link classification, failures/advice, result helpers | filesystem discovery, Tauri invoke, UI state |
| S2 Rust discovery | selected-root containment, deterministic walk, symlink rejection, disk reads, UTF-8 check, budgets, cancellation, partial discovery payload | YAML meaning, product advice copy, Markdown rendering |
| S3 TypeScript UI | explicit command/context entry, invoke lifecycle, prior-result preservation, localized summary/list/findings, existing-tab open | writes, background refresh, graph model, filesystem bypass |

Do not duplicate YAML conformance in Rust and TypeScript. Rust reports bytes,
paths, unreadable/limit state, and cancellation; the TypeScript model determines
OKF meaning.

## Slice Order

### S0 — Contract And Fixtures

- Pin commit `ee67a5c`, this matrix, link algorithm, budgets, and result shape.
- Include fixtures for a valid root-versioned bundle, versionless root index,
  nested index with frontmatter, reserved files containing `type`, empty/index
  advice, log date/order advice, invalid/unclosed YAML, missing/empty/numeric/list
  `type`, unknown fields/types, broken inline links, external links, bundle-root
  links, extensionless/directory targets, and a `../` containment escape.
- Include unreadable/non-UTF-8 and every budget/cancel boundary at the proper
  discovery layer.
- Include a small UTF-8 Japanese multi-file novel/essay bundle so the fixture
  reflects Hazakura's real writing use case.
- Do not vendor or copy the full upstream spec.

### S1 — Pure OKF Model

- Add the side-effect-free TypeScript parser/validator and thin result model.
- Keep unknown frontmatter data tolerable and source-preserving.
- Parse frontmatter with the direct `yaml` dependency; never claim conformance
  from delimiter detection alone.
- Do not connect filesystem discovery or UI.

### S2 — Explicit Bounded Discovery

- Add one cancellable async Rust command that arms cancel before
  `spawn_blocking` so `cancel_okf_bundle_scan` cannot race worker start.
  Reject concurrent scans in the cancel store; always disarm after the walk.
- Bound OS reads with `File::open` + `take(limit + 1)` so a growing file cannot
  allocate past the remaining file/total budget. Count actual bytes read toward
  `total_bytes_read` before UTF-8 decode.
- Reject symlink bundle roots before canonicalize.
- Reuse existing path/search guards where their contract matches; pin the five
  OKF-specific budgets above and return deterministic partial state.
- No startup scan, background indexing, cache database, or automatic refresh.

### S3 — OKF Review Surface (Feature Complete)

- Show the read-only disk-snapshot summary, file list, findings, dirty-tab note,
  partial/error states, and concept-open action in one lightweight surface.
- Keep keyboard access, Japanese / kana / English copy parity, narrow-window
  behavior, no-Markdown-render boundary, and main-window-only invocation explicit.
- Localize finding descriptions instead of exposing model-internal English
  messages, give repeated open controls path-specific accessible names, and
  close/cancel the surface when the owning workspace changes.
- Source tests plus a representative packaged-app OKF flow must pass before the
  feature is called complete.
- Do not claim release readiness from S3 alone.

### S4 — Distribution Confidence And Release Decision (Ship Ready)

- Run focused OKF fixtures and normal frontend / Rust / build gates.
- Exercise representative valid, invalid, partial, cancelled, containment,
  dirty-tab, link-advice, concept-open, and rerun flows in the packaged app.
- Fold the remaining v1.10 breadth into the packaged/signed matrix: IME, Save
  As, recovery, e-book/EPUB alignment, accessibility, sandbox, helpers, long
  document, and failure status.
- Only after this gate may a separate release slice bump to `1.11.0`, prepare
  App Store copy/package, upload, submit, or publish.

## Safety And Path Boundary

- Read only the explicitly selected bundle root and regular Markdown files
  inside it; never follow symlinks or normalized escapes.
- Never execute project files, inspect Git state, fetch external links, or fetch
  frontmatter `resource` / citation URLs.
- Never render bundle Markdown, inline HTML, or images in the review surface.
- Keep the command in the main Safe Editor window; Agent Workbench and Assist
  cannot invoke it.
- No background refresh, file watcher, network request, AI indexing, semantic
  embedding, or startup analysis.
- No automatic fix, save, manifest, cache database, or hidden persistent graph.

## Explicit Non-goals For v1.11

- treating all Markdown workspaces as OKF automatically;
- generating or rewriting `index.md`, `log.md`, frontmatter, links, or types;
- chapter ordering, multi-file move / rename / delete, or batch Save;
- persistent Book manifest, hidden database, or saved link graph;
- whole-book Preview, e-book Mode, PDF, or EPUB export;
- graph visualization, semantic search, embeddings, RAG, or AI enrichment;
- Git, LSP, terminal, plugins, arbitrary command execution, or release
  automation inside the app.

## v2 Boundary

v2 remains the first lane that may treat an explicit set of Markdown files as
one book. It may define visible, reversible Hazakura conventions for chapter
order and whole-book reading/export, but it must not claim that base OKF defines
those semantics. The v1.11 compatibility model may be reused as input, while
Book Scope stays a separate product contract.

# v1.1-v1.2 Follow-up

Status: Operational
Scope: User-observed follow-up after the v1.0 Golden Manuscript smoke
Authority: High
Last reviewed: 2026-06-29

## Decision

The 2026-06-28 user-side pre-v1 checklist supports a v1.0 Go decision.
The checklist is evidence, not a requirement that every box be checked.
The decisive No-Go conditions were not observed: the app launches,
source was not lost, long-form e-book reading remained usable, EPUB
export and page breaks worked in Apple Books, Local Assist applied to the
intended document through explicit review, and the App Store lane kept
its safety boundary.

`1.0.0` was subsequently approved and released on the Mac App Store.
No hotfix blocker has been reported. The canonical listing packet remains
`docs/releases/1.0.0-app-store-listing-copy.md`; the published v1
artifacts are immutable, and the observations below now form the active
v1.1-v1.2 intake queue.

## v1.1 Kickoff

Start with **Reader / editor position ownership** and **Preview / tab
position persistence** as one bounded continuity investigation. Reproduce
each transition independently, pin current behavior, and change only the
smallest shared ownership rule. The intended mental model is:

- explicit entry from the editor or Preview uses the current visible
  source position;
- a passive reader remount preserves the active reader location;
- `編集に戻る` returns through the active reader source estimate;
- Preview reopen and tab/external-link transitions do not silently reset
  an unrelated editor position.

Take forced-termination Recovery next unless a source-loss signal makes
it a hotfix. Image-heavy layout, Diff wording, menu containment, and
backup-file investigation follow one case at a time. Keep source/package
metadata at `1.0.0` until the first v1.1 change is accepted and a
candidate lane is deliberately opened.

## v1.1 Position Continuity: Current Behavior Pins

The first continuity slice began reproducing the five transitions above
in focused tests, before any behavior change. Not all five are pinned
yet. Progress as of 2026-06-29:

1. **Explicit e-book entry** (editor -> e-book Mode). Pinned: when the
   side pane switches to e-book Mode and a visible editor scroll heading
   exists, the reader opens at that heading's chapter. Already covered
   by `AppWorkspace.test.tsx` "opens the e-book pane near the current
   editor heading" and "uses the visible editor scroll heading".
   Matches the intended mental model; no change needed.
2. **Passive reader remount.** Pinned (reader side only): the stored
   `ebookLocation` restores for an unmoved tab (`AppWorkspace.test.tsx`
   "returns to an unmoved tab restores its reader location"). However,
   `ebookLocation` is a **single slot keyed by `documentKey`**, not a
   per-tab Map. When the active tab changes to B and B's reader moves,
   the single slot holds B's location, so returning to A loses A's last
   reader position and A falls back to the editor heading anchor.
   Pinned as "single-slot limitation: A location is lost once B moves".
   Gap comment added at `AppWorkspace.tsx` `ebookLocation` state.
   NOTE: the existing tab tests assert the *reader* location only; they
   do not yet verify that an unrelated *editor* position survives a tab
   switch. Add that assertion before declaring transition #5 pinned.
3. **`編集に戻る` (return to edit).** Pinned: closes Reading Focus and
   moves the editor to the reader source estimate (`sourceLine` if
   available, else the chapter heading), with editor focus. Already
   covered by "returns from Reading Focus to the active reader chapter
   heading" and "...to an estimated reader source line". Matches the
   intended mental model; no change needed.
4. **Preview reopen (top-reset).** Pinned as an ownership-boundary test
   (`SidePane.test.tsx` "keeps the shared scroll container (and its
   scrollTop) across a side-pane mode switch in jsdom"). Finding: the
   scroll container is the **shared SidePane wrapper div** (`previewPaneRef`),
   and SidePane renders this outer div in every mode, swapping only the
   inner leaf (PreviewPane / EBookPane / DiffPane). So the div identity
   and its `scrollTop` persist across a mode switch in jsdom — the naive
   "PreviewPane state was lost on unmount" model is wrong. The real-world
   "Preview reopen starts at the top" symptom must therefore come from
   something else (e.g. the rendered HTML being replaced and the scroll
   height collapsing, or the editor-sync path resetting `scrollTop`), and
   still needs real-layout reproduction. A future save/restore contract
   keyed by document identity belongs at SidePane / AppWorkspace, not the
   PreviewPane leaf. Code comment at `PreviewPane.tsx` records the owner.
5. **Tab / external-link transitions resetting the editor.** Pinned (tab
   side): `EditorPane.test.tsx` "resets the cursor when the documentKey
   changes (tab switch)" shows a `documentKey` change remounts the
   CodeMirror editor, so the previous tab's cursor/scroll position is
   lost (resets to document start), and returning to the first tab also
   remounts from the top. This is the editor-side analogue of the reader
   single-slot limitation (#2). External-link-driven resets are **not**
   reproduced yet; take them as the next bounded observation (or
   `verified no-op`) before that sub-transition is considered pinned.

The reader-side pin tests are grouped under "v1.1 position-continuity
pins" in `AppWorkspace.test.tsx`; the editor-side pin is in
`EditorPane.test.tsx`; the Preview scroll-container pin is in
`SidePane.test.tsx`. All intentionally make no behavior change. With #2,
#4, and #5(tab) pinned, the single-slot limitation (#2) and the
editor remount reset (#5 tab) are now the two concrete ownership gaps a
future change can target. External-link reset (#5 link) remains a
separate slice.

## v1.1: Daily-use Continuity And Trust

Prioritize issues that can make a writer lose position, confidence, or
the meaning of a review action.

1. **Reader / editor position ownership.** After e-book Mode has moved,
   reopening it may prefer stored reader position over the current editor
   position. Define an understandable rule for explicit editor entry,
   stored reader continuation, and `編集に戻る`.
2. **Preview and tab position persistence.** Preview can reopen at the
   top, and external-link or tab transitions may reset the editor
   position. Reproduce each transition independently before changing the
   shared position bridge.
3. **Recovery availability after forced termination.** Reproduce with a
   local folder and a Google Drive folder separately. Record whether the
   backup had time to be written, whether the workspace bookmark was
   restored, and whether reopening the same document suppresses the
   candidate. No source loss was observed, so this is a focused
   reliability investigation rather than a v1 blocker.
4. **Image-heavy layout edge cases.** Cover consecutive images in e-book
   Mode, a first-line or nested-path image in L Mode, and PDF page breaks
   through a large image. Preserve the passing ordinary image/text case.
5. **Diff action clarity.** Evaluate changing the floating comparison
   actions from a close-like affordance to explicit `採用` / `破棄`, and
   keep dirty/save state visible when the file-comparison surface is open.
6. **Context-menu viewport containment.** Reposition or flip the
   right-click menu when it would extend below the window.
7. **Unexpected backup files in Trash.** Determine whether recurring
   `.bak` files come from Hazakura, a cloud-sync interaction, or another
   agent/machine before changing backup cleanup behavior.

## v1.2: Polish And Expectation Setting

1. **Local Assist quality.** Collect short examples of unclear output,
   then prefer one small prompt adjustment at a time. Keep `プレビュー`,
   output variability, explicit Diff review, and no-auto-save wording.
2. **Export UX.** Align EPUB dialog button order with the other save
   dialogs, add an in-app explanation that e-book Mode and exported EPUB
   are related but not identical, and state that Hazakura is not a full
   EPUB production tool.
3. **HTML image-size limit.** Re-evaluate the 10 MB editing/export limit
   using image-heavy fixtures before raising it; keep memory and binary
   safeguards explicit.
4. **L Mode task checkboxes.** Decide whether source-first display should
   permit direct checkbox toggling and make the resulting rule clear.
5. **Command discovery.** Align the right-click insertion menu with
   `Cmd+Shift+P` where the same safe actions already exist; do not create
   a second command system.
6. **Empty-state composition.** Revisit logo placement only after the v1
   App Store screenshot set exposes a concrete imbalance.
7. **Experimental workspace drag/drop.** Either make the existing bounded
   move reliable or keep it de-emphasized; do not broaden it into a full
   file manager.

## Accepted Limits And Optional Proof

- 100k-character documents are not a comfort guarantee. The priority is
  no crash, source loss, or silent failure; measure before optimizing.
- Save As may reset scroll position and undo history until the editor
  session id is separated from the document key.
- PDF margin and page-size controls, advanced EPUB metadata, cover and
  navigation editing, multi-file books, and vertical writing remain
  post-v1 work.
- Apple Books proof is sufficient for v1. Kindle Previewer remains an
  optional compatibility check, not a release gate.
- Local Assist remains a preview companion, not a strong general-purpose
  AI editor.

## Intake Rule

Take one reproducible item at a time. Record the document shape,
workspace type, exact transition, expected result, and observed result.
Do not bundle unrelated visual cleanup or refactoring with a fix. A
verified no-op or accepted limitation is a valid outcome.

## B-1: Review Desk is a top-level review surface, not a `RightPaneMode` value

### Decision

Add a new top-level state `ReviewSurface` (a nullable string union) in
addition to the existing `RightPaneMode`. `RightPaneMode` is not
extended. `ReviewSurface` is a separate state slot because Review Desk
intentionally replaces the editor area, while the four existing
`RightPaneMode` values live next to the editor.

```ts
// Planned addition to src/types.ts
export type ReviewSurface =
  | "review"        // open for AI candidate or change review
  | null;           // editor area visible, no review surface

// Existing, unchanged
export type RightPaneMode = "preview" | "compare" | "outline" | "agent";
```

The new state lives next to the existing `rightPaneMode` in a hook
named `useReviewSurfaceState` (mirroring `useAppViewState.ts:1-15`).
It is owned by the same `useAppViewState` module surface so the App
composition stays a single call.

### Why not a fifth `RightPaneMode` value

The Diff path today is `rightPaneMode = "compare"` and it already
hides the editor area, sharing the main content width. That worked
because Diff is a transient "I need to compare two things" action.

Review Desk is different:

- It is a deliberate, named surface in the product narrative.
- It hosts multiple internal sub-states (candidate, change review,
  draft review, conflict review) that are not all the same render.
- It has its own keyboard entry (`Cmd+Shift+R`, see B-3) and its own
  branding / About text.
- Adding it to `RightPaneMode` would make the side-pane
  `useSidePaneState` and `useSidePaneController` carry logic they
  were never designed for, and it would break the type
  `RightPaneMode | null` that several hooks already destructure
  (e.g. `src/hooks/useSidePaneState.ts:5`,
  `src/hooks/useSidePaneController.ts:6`).

### What stays the same

- Diff still uses `rightPaneMode = "compare"`. Its existing behavior
  (hide editor, full-width compare) is preserved.
- Preview, Outline, and Agent continue to live as `rightPaneMode`
  values.
- `useSidePaneToggles.ts` is not touched.
- `useAppViewState.ts:1-15` keeps `rightPaneMode` and `zenMode`; a
  new `reviewSurface` field is added next to them in the same return.

### Files the future implementation slice will touch

- `src/types.ts:77` ŌĆö add `ReviewSurface` type.
- `src/hooks/useAppViewState.ts:1-15` ŌĆö add `reviewSurface` state.
- `src/App.tsx` (root JSX at line ~648) ŌĆö render the review surface
  instead of the editor area when `reviewSurface !== null`.
- (New) `src/components/ReviewSurface.tsx` ŌĆö host for the AI
  candidate / change review sub-states. Skeleton only; this slice
  does not build it.

## B-2: `CompareViewState` becomes a discriminated union before AI lands

### Decision

Today `CompareViewState` is a single flat object in
`src/types.ts:226-237` with `kind: "file" | "changes"`. That works
because all four "review-shaped" cases (file-to-file, editor-vs-disk,
draft-vs-disk, conflict-vs-disk) share the same line structure.

When the AI candidate case lands, it has a different shape: there is
no `leftPath` (the user has not yet accepted anything), the right
side carries a "candidate provenance" header, and the "apply" button
becomes a first-class state transition rather than a side effect.

To keep the renderer (`src/components/DiffPane.tsx:1-356`) from
growing past maintainability, the union is split before the first AI
slice lands.

```ts
// Planned shape (illustrative; not yet implemented)

// The "what am I looking at" ŌĆö owned by the controller
export type CompareCase =
  | {
      kind: "file";
      key: string;
      leftPath: string;
      rightPath: string;
      anchor: { path: string; label: string };
      target: { path: string; label: string };
    }
  | {
      kind: "changes";
      key: string;
      scope: "buffer-vs-disk" | "draft-vs-disk" | "conflict-vs-disk";
      documentPath: string;
      documentLabel: string;
    }
  | {
      kind: "candidate";
      key: string;
      documentPath: string;
      documentLabel: string;
      candidateId: string;
      candidateSource: "selected-text" | "full-document" | "preset";
      instructionLabel: string;
    };

// The "what is on screen" ŌĆö owned by the diff engine
export type CompareViewState = {
  caseKey: string;
  lines: DiffLine[];          // existing shape, unchanged
  additions: number;
  removals: number;
  // No kind, no leftPath/rightPath. The renderer asks the
  // controller to recover the CompareCase by caseKey.
};
```

The renderer (`DiffPane.tsx`) stops reading `leftPath` / `rightName`
directly. It renders the diff body and asks the controller for the
`CompareCase` to show labels and the action bar. Each case gets a
dedicated subcomponent (`<FileCompareView>`, `<ChangeReviewView>`,
later `<AiReviewView>`), as foreshadowed in gate finding R-2.

### Why split now, not when the AI slice lands

The current `CompareViewState` is consumed in five places:

- `src/hooks/useCompareExecution.ts:11, 26, 57, 90, 179` ŌĆö produces
  it for all four review-shaped cases.
- `src/hooks/useCompareSetupActions.ts` ŌĆö reads the current
  `compareView` to decide whether to overwrite.
- `src/hooks/useCompareState.ts` ŌĆö owns the slot.
- `src/components/DiffPane.tsx` ŌĆö renders it.
- `src/types.ts:226-237` ŌĆö defines it.

Splitting the case (source) from the view (render) before AI lands
keeps the AI slice mechanical: add a `kind: "candidate"` case, add
`<AiReviewView>`, done. Without the split, the AI slice would either
force-grow `CompareViewState` or shove the new shape into a parallel
slot, both of which the gate flagged as a structural risk.

### What stays the same

- `DiffLine` and `DiffSplitRow` types are unchanged.
- The four existing "changes" cases continue to round-trip through
  the same Rust compare engine. No backend change.
- `useCompareController` (composition root) keeps its current API
  surface; the new `CompareCase` is internal.

### Files the future implementation slice will touch

- `src/types.ts:226-237` ŌĆö add `CompareCase`, refactor
  `CompareViewState`.
- `src/hooks/useCompareExecution.ts` ŌĆö produce both `CompareCase`
  and `CompareViewState` together.
- `src/hooks/useCompareController.ts` ŌĆö expose
  `getCompareCaseByKey(caseKey)`.
- `src/components/DiffPane.tsx` ŌĆö switch on case kind, render the
  right subcomponent.
- (New) `src/components/FileCompareView.tsx`,
  `src/components/ChangeReviewView.tsx` ŌĆö extracted from `DiffPane`.

## B-3: Reserve `Cmd+Shift+R` for the Review Desk entry point

### Decision

`src/hooks/useGlobalKeyboardShortcuts.ts:1-269` does not currently
register a `Cmd+Shift+R` handler. The shortcut will be reserved for
the Review Desk entry point.

The reservation is a one-line no-op handler that:

1. Listens for `Cmd+Shift+R` (and `Ctrl+Shift+R` on non-mac).
2. Calls `event.preventDefault()` to stop the platform default
   (browser reload, in the WebView).
3. Does not yet trigger any review surface open/close behavior.
4. Is commented with a pointer to this document so the future
   Review Desk UI slice knows where to attach the toggle.

The reservation ships without user-visible behavior. It is the
smallest possible slice to claim the shortcut.

```ts
// Planned addition to useGlobalKeyboardShortcuts.ts
// Reserved for the Review Desk entry point. See
// docs/reviews/v0.7-review-desk-design-decisions.md (B-3).
if (matchesCommandShiftR(event)) {
  event.preventDefault();
  // Toggle is intentionally not wired yet.
  return;
}
```

### Why reserve before the UI slice

- `useGlobalKeyboardShortcuts.ts` is 269 lines and is the
  catch-all for "I pressed something". Any future Review Desk
  command (open, close, jump to next change) will live here.
- `Cmd+Shift+R` is the conventional "review" shortcut in editors
  (VS Code, JetBrains, Sublime). Claiming it first prevents a
  accidental claim by another surface.
- Reserving a no-op is a one-line, zero-risk change. The
  reservation is testable: a smoke check that pressing
  `Cmd+Shift+R` does not trigger the WebView reload.

### What stays the same

- All other `Cmd+Shift+*` shortcuts (`Cmd+Shift+T` for Insert Table,
  `Cmd+Shift+W` for window close, `Cmd+Shift+F` planned for Global
  Search in a later slice) keep their current bindings.
- macOS View menu items are not changed in this slice.
- The reservation does not alter the `RightPaneMode` state or
  introduce `ReviewSurface`. Those are B-1.

### Files the future implementation slice will touch

- `src/hooks/useGlobalKeyboardShortcuts.ts` ŌĆö add the no-op
  reservation handler.
- `docs/smoke-checklist.md` ŌĆö add a smoke row for
  "Cmd+Shift+R does not reload the WebView".

## Combined Effect on the v0.7 Implementation Order

The three decisions together imply this revised order for the v0.7
slices, building on the order already stated in
`docs/reviews/v0.7-readiness-gate.md`:

1. **B-1 + B-2 + B-3 docs (this slice, docs-only)** ŌĆö done.
2. **F-1 implementation** ŌĆö split
   `useAgentWorkbenchSessionActions` into
   `useAgentSessionLifecycle` + `useAgentLaunchGate`. No surface
   changes, no new state, behavior-preserving.
3. **B-3 reservation** ŌĆö wire the no-op `Cmd+Shift+R` handler. One
   line, plus a smoke row.
4. **B-1 wiring** ŌĆö add `ReviewSurface` state to
   `useAppViewState`. App.tsx JSX adds a single conditional
   rendering slot. Still no feature.
5. **B-2 wiring** ŌĆö refactor `CompareViewState` into the
   `CompareCase` + `CompareViewState` split. Extract
   `<FileCompareView>` and `<ChangeReviewView>` from `DiffPane.tsx`.
   No new case yet.
6. **First Review Desk feature** ŌĆö `kind: "candidate"` case +
   `<AiReviewView>`. Only after B-1 and B-2 are landed.

Steps 3, 4, and 5 are each independent small slices. They can be
taken in any order but each is small enough to be reviewed in
isolation. Step 6 is the actual AI candidate -> Diff -> explicit
apply slice, which has its own design review and is out of scope for
this decision doc.

## Open Questions

- Q-1. Should the `ReviewSurface` type also accept a
  "change-review-only" sub-mode that is not the AI candidate
  (e.g. manually opened editor-vs-disk) so that B-1 also subsumes
  the "Diff" toggle eventually? The current decision keeps Diff
  on `rightPaneMode = "compare"` for stability and answers this
  in a later slice.
- Q-2. The `CompareCase.key` is described as a string. The
  concrete generation rule (path hash + revision) is left to the
  implementation slice. It needs to be stable enough for a
  review to be re-resumed after restart.
- Q-3. The reservation handler in B-3 will eventually need to
  coexist with a "next change" / "previous change" navigation
  pair. Whether those are `Cmd+Shift+]` / `Cmd+Shift+[` or
  something else is a future decision; not in scope here.

# v0.7 Review Desk Readiness Gate

Status: Review
Scope: Read-only structural review before v0.7 Review Desk implementation
Authority: Medium (gate artifact, not canonical spec)
Last reviewed: 2026-06-01

## Purpose

Per `docs/roadmap.md:254-263` and `docs/next-goals.md:8-46`, v0.7 Review Desk
feature work must not start until this gate has been run. The gate is
read-only unless the user explicitly approves a follow-up cleanup slice.

This document inventories the responsibilities that still live in
`src/App.tsx`, the feature boundary health of the `src/hooks/` and
`src/components/` graphs, the module split status of
`src-tauri/src/lib.rs`, and the Diff / Review / Tabs / FileTree /
Agent Workbench / Preferences surfaces, then classifies each finding into
one of five buckets.

The gate deliberately does not propose new product behavior. The
non-goals from `docs/next-goals.md:39-45` still apply: no Review Desk
UI, no AI candidate -> Diff -> explicit apply, no Git / LSP / terminal
/ plugin / indexing, and no Agent auto-apply or session restore.

## Surface Inventory

### `src/App.tsx` (867 lines)

Composition vs rendering split:

- Lines `47-647` (~600 lines): hook composition. The App function
  destructures state and handlers from 23 hooks. There are no direct
  `useState`, `useEffect`, `useMemo`, or `useCallback` calls inside App.
  This part is the orchestration layer, not state ownership.
- Lines `648-867` (~220 lines): the root JSX tree. It mounts
  `SakuraPetals`, `AppTopChrome`, `AppWorkspace`, `AppStatusBar`, and
  the overlay cluster (`AppOverlays` + `CloseDialogs` + `QuickOpen` +
  `PreferencesDialog` + `WorkspaceContextMenu`) inline rather than

## Findings

### F. Fix now (gate can complete this)

- **F-1. `useAgentWorkbenchSessionActions.ts` is too dense.** 309 lines,
  six distinct handler groups: `refreshAgentSessionState` (line 68),
  `sendWorkspacePathToAgent` (line 112), `checkAgentLaunchGate`
  (line 158), `stopAgentSession` (line 257), `requestAgentLaunchGateCheck`
  (line 295), `requestAgentSessionStop` (line 299). Review Desk will
  add a seventh (selected-text -> candidate send). Before that slice,
  split into `useAgentSessionLifecycle` (refresh / stop) and
  `useAgentLaunchGate` (gate check / request). Same composition
  pattern as `useCompareController` already uses (composes
  `useCompareSetupActions` + `useCompareExecution`).
  Estimated slice: ~40 lines of glue + a new test that exercises the
  two halves independently.

- **F-2. App.tsx still composes the root JSX tree inline.** Lines
  648-867 render `<main>` with `SakuraPetals`, `AppTopChrome`,
  `AppWorkspace`, `AppStatusBar`, and the overlay cluster by hand.
  Extracting a single `<AppShell>` that owns these five mounts makes
  the App function a pure hook composition file. Estimated slice:
  ~30 lines of net new (new component + App.tsx shrink).

### B. Before v0.7 implementation

- **B-1. Review Desk needs a dedicated right-pane mode value.** Today
  `RightPaneMode` is a string union ("preview" / "diff" / "outline" /
  "agent") owned by `src/hooks/useAppViewState.ts:6` and switched via
  `src/hooks/useSidePaneController.ts`. Diff is a side pane. A formal
  Review Desk surface that hosts AI candidate -> Diff, current
  editor-vs-disk, draft vs disk, and file-to-file would benefit from
  its own mode (e.g. "review") rather than overloading "diff". Decide
  this before the first Review Desk slice.

- **B-2. `CompareViewState` is a flat two-case union.** Defined in
  `src/types.ts:226-234` with `kind: "file" | "changes"`. Today that
  is enough because the four "review-shaped" cases (file-to-file,
  editor-vs-disk, draft-vs-disk, conflict-vs-disk) all use the same
  `DiffLine[]` body. Adding an AI candidate case will need a third
  `kind` plus a way to attach the candidate provenance. Either add the
  case now or refactor to a discriminated union of `CompareCase`
  before the first AI review slice.

- **B-3. Reserve `Cmd+Shift+R` for the Review Desk entry point.**
  `src/hooks/useGlobalKeyboardShortcuts.ts` (269 lines) does not
  register a Cmd+Shift+R handler today. The same shortcut is widely
  expected to mean "Review" in editors. Reserve it before any other
  slice accidentally claims it.

- **B-4. Three Compare hooks split already exists; replicate it.**
  `useCompareController` (22 lines, `src/hooks/useCompareController.ts`)
  composes `useCompareSetupActions` (207 lines) +
  `useCompareExecution` (249 lines). This is the right shape for any
  hook that owns both a setup phase and an execute/review phase. Apply
  the same pattern when extracting the agent session hook (see F-1).

### V. v0.8+ (Product Preview, later)

- **V-1. `EditorPane.tsx` (931 lines) mixes view lifecycle with
  Markdown helpers.** It owns CodeMirror 6 `EditorView` construction,
  theme / wrap / invisibles / tab-size / spellcheck compartments, and
  the Markdown insertion helpers (bold, italic, code, link, table,
  headings) inline. Extracting a pure `markdownInsertion.ts` helper
  module would let `EditorPane.tsx` shrink below 700 lines and let
  the Markdown insertion path be unit-tested without mounting the
  editor. Not required for v0.7; defer until the Markdown toolbar
  slice in v0.8.

- **V-2. `src-tauri/src/lib.rs` (990 lines) can split its
  `#[tauri::command]` wrappers by feature.** Today the wrappers and
  the `_with_store` glue live together. The `auto_backup` pattern
  (impl in module, thin wrapper in lib.rs) is the gold standard.
  Migrating the other 21 commands to the same shape — `commands/`
  submodule per feature (`file_io`, `workspace`, `image`, `print`,
  `menu`, `opened_files`, `agent`) — would shrink `lib.rs` below 400
  lines. The agent commands are the trickiest because they share the
  `AgentWorkbenchSessionStore` and have `_with_store` variants; the
  refactor needs to land alongside an adapter extraction.

- **V-3. Rust `lib.rs` has 10 `use` statements and only 4 module
  re-exports.** `pub(crate) mod` declarations for `types`, `util`,
  `agent`, `menu`, `auto_backup` live at the top of `lib.rs:8-16`.
  `tests` is gated. This is fine, but once V-2 lands a `commands/`
  tree, the existing flat module list will want to be reorganized
  into `domain/`, `commands/`, and `runtime/` siblings.

### T. Fold back together (split went too far)

- **T-1. The 19 sub-25-line "state-pair" hooks can fold into a smaller
  set.** Each one is a `useState` (or two) plus a setter. The cost
  is not the code; it is the "which hook owns X" lookup during code
  review. Candidate merges:
  - `useAppEditorRefs` (18) + `useAppDialogRefs` (76) ->
    `useAppDomRefs` (refs are not state, they belong together).
  - `useEditorSelectionState` (16) + `useEditorTabsState` (14) ->
    `useEditorTabsState` (selection info is per-tab-derived).
  - `useThemeMenuStateSync` (11) -> inline into `useAppMenuIntegration`
    (it is one effect, no state of its own).
  - `useAppFeedbackState` (13) + `useDraftRecoveryState` (11) +
    `useDocumentSafetyActions` (21) + `useRecoveryActions` ->
    co-locate as `useRecoveryFlow`. All four are about
    conflict / draft / save-error recovery messages and would be
    easier to read side by side.
  - `useQuickOpenState` (19) -> fold into the planned
    `useCommandPalette` slice in v0.7, since both own the same "open
    launcher" state.

- **T-2. `useWorkspaceContextMenu` + `useWorkspaceContextMenuDismissal`
  should be one hook.** Both are 30-50 lines and only run together.
  Today the dismissal hook only exists so a separate effect can
  import it. Re-merge before the v0.7 implementation phase.

- **T-3. `useWorkspaceStatePersistence` and `useWorkspaceRestore` are
  sibling concerns.** Persistence writes; restore reads. They could
  live as one `useWorkspaceStateLifecycle` with two methods. This
  matches the compare composition pattern in B-4.

### R. Review Desk structural risk

- **R-1. The "review" surface is currently a side pane, not a mode.**
  All four existing "review" cases (file-to-file, editor-vs-disk,
  draft-vs-disk, conflict-vs-disk) share `rightPaneMode = "diff"`. If
  Review Desk lands as another value of `rightPaneMode`, the
  SidePane state will be over-stretched. A cleaner shape is:
  - Keep `rightPaneMode` for the three persistent companions
    (preview, outline, agent).
  - Make Diff / Review a first-class "review surface" that
    intentionally replaces the editor area, similar to how the
    current diff toggle already hides the editor and uses full width
    (per `docs/current-status.md`). Reserve that pattern for Review
    Desk.

- **R-2. `DiffPane.tsx` (356 lines) assumes the four existing
  cases.** Adding an AI candidate case risks growing this component
  past 500 lines unless the case fan-out is moved into a router
  (e.g. `case "file" -> <FileCompareView>`, `case "changes" ->
  <ChangeReviewView>`, future `case "candidate" -> <AiReviewView>`).
  Plan this split before the first AI slice.

- **R-3. AI candidate -> Diff -> explicit apply needs a persisted
  decision log location.** `docs/authoring-feature-readiness.md` and
  the earlier `チカちゃん` review both identified that Review Desk
  must keep a "review decision" trail. The structural decision is:
  store the log under `.hazakura/reviews/` inside the selected
  workspace (per-workspace, reproducible) or in app-local storage
  (per-device, not committable). This gate cannot answer it, but
  flags it as the first open design question once v0.7 work starts.

- **R-4. `app-state` boundary for Review Desk does not exist yet.**
  Currently `App.tsx` owns the `compareView` state via
  `useCompareState`. If Review Desk also needs to own a "pending
  candidate" state and a "review decision log" state, App.tsx will
  grow again. The clean answer is a `useReviewDeskState` hook that
  owns `{ pendingCandidate, reviewCases, decisions }` and a single
  `useReviewDeskController` for actions, mirroring the Compare
  pattern.

## Suggested Sequencing

This gate does not approve any of the above. The sequence implied by
the buckets:

1. F-1 split `useAgentWorkbenchSessionActions`. F-2 extract
   `<AppShell>`. These are small mechanical slices with no
   boundary impact.
2. B-1 / B-2 / B-3 / B-4 design Review Desk state, type, and
   shortcut reservation. These are decisions, not code, and can
   land as a single v0.7 kickoff doc.
3. T-1 / T-2 / T-3 fold the small hooks back together while
   touching the area for F-1 and B decisions, so the merges
   ride along rather than block.
4. R-1 / R-2 / R-3 / R-4 are the actual Review Desk surface
   design. Land these after F + B + T, before any feature code.

## Out of Scope for This Gate

The gate explicitly does not approve:

- New `#[tauri::command]` or new frontend commands.
- A new Review Desk surface implementation.
- A command palette, global search, or settings consolidation.
- Any change to `RightPaneMode`, `CompareViewState`, or `App.tsx`
  rendering.
- Any docs change beyond this gate artifact and the
  `docs/current-status.md` entry recording gate completion.

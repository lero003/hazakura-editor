# v1.13+ Refinement Roadmap

Status: Active development direction
Scope: Post-v1.12 product refinement through try-and-error before v2 Book Scope
Authority: High for post-v1.12 sequencing; each experiment still needs a focused slice contract
Last reviewed: 2026-07-18 (v1.13 published; v1.14+ refinement open)

## Decision

v1.12（OKF scaffold までを届けた橋渡し）は Mac App Store で公開済み
（user-reported 2026-07-17）。その先の v1.x は、**固定の大規模フェーズ列車ではなく、
試して残す refinement 期**とする。

- v1.13 Interaction Clarity の 7 スライスは **最初の実験候補プール**であり、必須の順番ではない。
- 2026-07-16 UX 提案（103 項目）は **観察ログと仮説の倉庫**であり、そのまま backlog にしない。
- 版番号は「検証済みの改善をまとめて出す箱」であり、機能約束ではない。
- v2 Book Scope は境界の拡大であり、試行錯誤で前倒ししない。

North star は変えない:

> Markdownで書き、本として読み、ローカルAIで整える。

Refinement 期の作業レンズ:

> 編集空間はひとつ。無反応をなくし、対象と状態を明示し、摩擦を小さく試して残す。

## Why Rethink

これまでの v1.8–v1.12 は、信頼 → 目的整理 → 単一文書構造 → OKF 点検 → ひな形、という
**順次ゲート**が有効だった。一方で post-v1.12 は:

1. 製品面はすでに多い（Preview / e-book / Reference / Outline / Diff / Assist / OKF …）。
2. 静的レビューは方向が正しいが、**実装済みとのズレ**や実機未確認が混ざる。
3. 103 項目を v1.13 / v1.14 / v1.15 に機械分割すると、再び「全部やる」圧力が戻る。
4. ユーザー意図として、**より良くできそうなことは小さく試してよい**。

したがって、順序固定のロードマップから、**仮説駆動の短い実験ループ**へ切り替える。

## Hard Rails (do not experiment away)

試行錯誤でも次は壊さない。違反しそうなら実験を止めて境界を見直す。

- Safe Editor が主面。IDE / Git / LSP / 汎用ターミナル / 拡張基盤にしない。
- Markdown / テキスト source が正本。黙った書き換え・自動適用・自動保存をしない。
- 編集可能な主バッファは一つ。Preview DOM 編集や二画面編集にしない。
- workspace 操作は選択 root 内。フルファイルマネージャにしない。
- background indexing、hidden document model、project-wide 自動構造解析をしない。
- Agent Workbench は Developer レーン専用の別 trust boundary のまま。
- 公開済み `1.12.0` レーンに無関係な polish を混ぜない（hotfix は再現と明示判断のみ）。
- 公開タグ・公開アセットは immutable。

## Experiment Loop

1 ラン = 1 仮説 = 検証可能な 1 スライス。

| Step | Do |
|---|---|
| 1. Observe | 実機摩擦、静的レビュー、status コピー、a11y、長文、失敗系から仮説を取る。 |
| 2. Revalidate | 現行ソースと照合し、すでに直っている指摘は捨てる。 |
| 3. Slice | 受入条件・非目標・戻し方を 1 段落で固定する。可能なら失敗テストを先に置く。 |
| 4. Try | 最小差分で実装。UI トーンと既存命名を尊重する。 |
| 5. Prove | focused test + 触った面の typecheck/test/build。ホスト依存は未実施と明示。 |
| 6. Decide | **Keep**（正本へ昇格）/ **Iterate**（小さく直す）/ **Revert or defer**（残さない）。 |
| 7. Ship box | 1〜数個の Keep をまとめて `v1.xx` 候補にできる。空の版は作らない。 |

「試す」は雑に広げることではない。**小さく試して、証拠で残すか捨てる**ことである。

### Selection Heuristics

優先して試す:

- 無反応・誤解・誤操作・未保存の見落としを減らす。
- 日本語 / かな / キーボード / IME / VoiceOver の到達性を直す。
- 既存導線の説明や状態表示だけで済む改善。
- 1 日〜数日で戻せる変更。

後回し / 設計契約が先:

- 上部 chrome の大規模再編、設定画面の全面再カテゴリ。
- whole-workspace Quick Open（depth/count/time/symlink 予算の設計が先）。
- 解析キャッシュ共有など「hidden model に見える」共有層。
- AI / OCR / Recovery 候補 UI の大規模統一。
- multi-file Book Scope、縦書き、manifest authoring。

明示的にやらない:

- background indexing / startup scan / auto-repair / auto-apply / auto-save
- provider 追加 UI、汎用 AI chat、任意コマンド実行
- Git / LSP / ターミナル / プラグイン

## Theme Pools (not fixed version numbers)

テーマは並行候補。版番号は後から付ける。

### Theme A — Signal & Silence（旧 v1.13 Clarity を含む）

無反応と誤解を減らす。最初の実験プールの中心。

候補（独立選択可）:

- Command availability と localized disabled reason
- Quick Open の scope / truncation を正直に書く（索引は別設計）
- 検索結果 open status のローカライズ
- Quick Settings の popover/dialog 意味論と focus return
- Preview / e-book の遅延 loading フィードバック
- OKF starter の progressive disclosure
- Workspace Trash の対象と disabled reason
- status message の種類と寿命の最小統一
- 破壊的 / 採用系コピーの「何が起きたか」明示

詳細の出発点: `docs/v1.13-interaction-clarity-plan.md`

### Theme B — Continuity & Orientation

再訪と「今どこにいるか」を滑らかにする。

候補:

- 最近使った workspace（上限付き・明示選択のみ）
- 同名タブの親フォルダ表示、タブオーバーフロー
- Reference の保持 / 隠す / 閉じるの visually explicit 状態
- 右ペイン共通ヘッダー（小さい契約から）
- 文書移動履歴の「戻る」が履歴だと分かる表現
- Start Panel の復旧要約と次アクション

### Theme C — Writing Loop Vocabulary

機能名より利用目的で見つける。**一度にシェルを作り直さない。**

候補:

- 目的語彙（書く・読む・確かめる・整える）を既存入口の説明・空状態・palette に足す
- Preview と電子書籍の違いを UI 内で短く説明する
- e-book → 編集位置へ戻る操作の発見性
- Outline 助言がエラーに見えないこと
- Diff / Assist / Import の「採用 ≠ 保存」の可視性
- 上部ボタンの目的グループ化は、**実機で摩擦が再現したときだけ**小さな契約で試す

### Theme D — Trust At The Edges

保存・復旧・書き出し・Assist の端を安心させる。

候補:

- 書き出し結果（出力先・成功/警告）の明示
- Export 進捗 / キャンセル可否の統一（既存境界内）
- Local Assist 利用可否とロック理由の短い可視化
- Import Assist 結果を「下書き」として強調
- エラー文の「起きたこと / 守られたこと / 次」テンプレへ寄せる
- 診断情報のコピー（秘密を増やさない範囲）

### Theme G — Media Boundaries（画像の同意付きゆるめ）

既定の「外部画像を勝手に読まない」は維持し、**設定・都度確認・書き出し時の実体化**で
Preview / 書き出しの摩擦を減らす。workspace 内相対パスは既に可。痛いのは
root 外ローカルとリモート URL。

候補（詳細契約: `docs/v1.xx-image-media-boundary-plan.md`）:

- **M0** ブロック理由と次アクション — **Keep**
- **M1** workspace 外ローカル同意（オフ / 都度 / 記憶）— **Keep**
- **M2** リモート画像 Preference（既定オフ、https のみ）— **Keep**
- **M3** 書き出し materialize（既定オン、source 非改変）— **Keep**
- **M4** assets 固定 rewrite + Command Palette — **Keep**

Hard rails for this theme: 起動時一括 fetch なし、OKF review に画像描画を足さない、
任意絶対パスの無制限 open なし、export の自己完結を Preview の一時表示より優先。

### Theme E — Structure & OKF Depth（still single-doc / bounded）

v2 の前に、すでに持っている構造と OKF 面を深める。

候補:

- OKF 点検の first-fix 1 件と入口説明
- ひな形作成前の短い内容説明
- Outline / structure advice のフィルタや Undo フィードバック
- pin 更新時の co-update 手順の運用（`docs/okf-spec-pin.md`）
- 配布版での構造・OKF・長文・復旧の横断証拠の積み増し

### Theme F — Distribution Confidence

機能ではなく証拠。公開判断の材料。

- TestFlight / 実機: IME、VoiceOver speech、narrow window、Reduce Motion
- 長文・budget・failure の残リスク記録
- App Store と Developer レーン差の sanity
- 申請中レーンを汚さない hotfix 判断

## v2 Entry Reminder

v2 に進む条件は、実験の成功数ではなく **境界を広げてよい証拠**である。

- 単一文書 structure model が安定している（v1.10 系）。
- OKF compatibility が bounded / explicit / read-only で説明できる（v1.11 系）。
- starter scaffold が明示操作として信頼できる（v1.12 系）。
- refinement で「無反応・状態誤解・配布信頼」の粗い穴が、Keep 済みか既知リスクとして記録されている。
- multi-file book が単一ファイルでは安全に届かない価値だと説明できる。
- Book Scope UI が候補リストではなく、少なくとも一つの方向に収束している。

## Working Rules For Agents

1. 公開済み `1.13.0` を無闇に触らない。hotfix は再現と明示判断が先。
2. 次の一手は「テーマを宣言して 1 スライス」だけ。パッケージ一括実装をしない。
3. 静的 UX 番号（UX-31 など）は参照 ID であってチケット義務ではない。
4. 実機で再現しない、またはすでに直っている指摘は Keep 候補にしない。
5. 実験が境界アラーム（indexing / auto-apply / 二画面編集 等）に触れたら即停止。
6. Keep したら `current-work` / `current-status` / 必要なら design を最小更新する。
7. 失敗した実験は恥ではない。revert 理由を handoff に 1 行残せばよい。
